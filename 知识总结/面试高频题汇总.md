## Java 核心（27题）

### Java基础（1题）

**Q1: Java中equals和==有什么区别？**

**标准答案**：
`==` 是运算符，比较规则分两种：
1. 基本类型比较的是值是否相等
2. 引用类型比较的是两个引用是否指向同一个对象

`equals()` 是 `Object` 的方法，默认实现本质上也是比较地址，但很多类会重写它来比较内容，比如 `String`、`Integer`。因此实际开发里，比较对象的业务内容是否相等，通常用 `equals`；判断是不是同一个对象实例，才用 `==`。

补充：
1. `String` 比较内容应该用 `equals`，不要依赖字符串常量池导致的 `== true`
2. 包装类型不要依赖 `==`，尤其要注意 `Integer` 的缓存范围 `-128~127`
3. 可能为 `null` 时，优先使用 `Objects.equals(a, b)`

**追问**：
- 为什么重写 `equals` 时通常还要一起重写 `hashCode`？
- `String a = "x"` 和 `String b = new String("x")` 用 `==`、`equals` 分别是什么结果？

### HashMap（5题）

**Q1: HashMap的底层实现原理？**

**标准答案**：
JDK 1.8 中，`HashMap` 的底层结构是 **数组 + 链表 + 红黑树**。

它的核心流程可以这样理解：
1. 先对 `key` 计算哈希值
2. 再通过 `(n - 1) & hash` 计算数组下标，定位到桶位置
3. 如果该桶为空，直接插入
4. 如果发生哈希冲突，就挂到该桶对应的链表上
5. 当链表长度达到 8 且数组长度至少为 64 时，链表会转成红黑树，降低查找成本

所以它本质上是用数组做快速定位，用链表或红黑树解决哈希冲突。

面试里可以补一句：
`HashMap` 在理想情况下查找复杂度接近 `O(1)`，但冲突严重时链表会退化成 `O(n)`，JDK 1.8 引入红黑树就是为了优化这种极端情况。

**追问**：
- 为什么是(n-1) & hash而不是hash % n？
- 为什么链表长度是8才转红黑树？

**Q2: JDK 1.7和1.8的HashMap有什么区别？**

**标准答案**：
1. 数据结构：1.7是数组+链表，1.8增加了红黑树
2. 插入方式：1.7是头插法，1.8是尾插法
3. 扩容优化：1.8通过`hash & oldCap`判断位置
4. 哈希算法：1.8简化了扰动函数

如果展开讲，最关键的是两点：
1. JDK 1.8 为了解决高冲突桶查询效率差的问题，引入了红黑树
2. JDK 1.8 把链表插入改成尾插法，配合新的迁移逻辑，避免了 JDK 1.7 多线程扩容时容易形成环形链表的问题

所以面试里讲这题，不要只背“数组+链表 vs 数组+链表+红黑树”，更重要的是把“为什么改”和“解决了什么问题”说出来。

**追问**：
- 为什么1.7用头插法会导致死循环？
- 1.8的尾插法如何避免死循环？

**Q3: HashMap的扩容机制？**

**标准答案**：
当元素数量超过 `容量 × 负载因子` 时会触发扩容，默认负载因子是 `0.75`。

扩容流程可以这样理解：
1. 新建一个更大的数组，通常容量变成原来的 2 倍
2. 把旧数组中的元素迁移到新数组
3. 迁移过程中重新确定元素在新数组中的位置

JDK 1.8 做了一个关键优化：
元素迁移后的位置只有两种可能：
1. 还在原来的位置
2. 原位置 + 旧容量

判断方式是看 `hash & oldCap` 是否为 0，这样就不用重新做一次复杂定位。

为什么要扩容？
因为随着元素增多，哈希冲突会越来越严重，查询和插入效率都会下降，所以需要通过扩容来降低桶中的冲突概率。

**追问**：
- 为什么负载因子是0.75？
- 扩容时如何保证线程安全？

**Q4: HashMap为什么线程不安全？**

**标准答案**：
1. JDK 1.7：扩容时使用头插法，多线程可能形成环形链表，导致死循环
2. JDK 1.8：虽然改成尾插法，但put操作不是原子的，多线程可能导致数据丢失

更本质地说，`HashMap` 线程不安全的原因是：
它内部没有任何并发控制，多个线程可以同时修改同一个桶、同一个链表，甚至同时扩容。

所以在并发场景下可能出现：
1. 两个线程同时 `put`，导致其中一次写入被覆盖
2. 扩容迁移时结构被破坏
3. 老版本中形成环形链表

所以面试里不要只说“因为会死循环”，那只是 JDK 1.7 的一个典型表现。更准确的说法是：
`HashMap` 的写操作和扩容操作都不是线程安全的，在并发修改场景下会导致数据不一致或结构异常。

**追问**：
- 如何让HashMap线程安全？
- ConcurrentHashMap和Hashtable的区别？

**Q5: 为什么链表长度达到8才转换成红黑树？**

**标准答案**：
根据泊松分布，链表长度达到8的概率极低（约0.00000006）。红黑树节点占用空间是链表的2倍，如果阈值太小会浪费空间；如果阈值太大链表查找效率低。8是性能和空间的平衡点。

另外还有一个容易漏掉的点：
不是只要链表长度到 8 就立刻树化，还要求数组长度至少达到 64。

原因是：
如果数组还很小，优先扩容通常比树化更划算，因为扩容后元素可能重新分散到不同桶里，冲突自然下降；只有当数组已经不小了，冲突仍然严重，才说明这个桶确实适合树化。

所以这个阈值设计本质上是在平衡：
1. 查找效率
2. 内存占用
3. 扩容成本

**追问**：
- 红黑树什么时候退化成链表？
- 为什么退化阈值是6而不是7？

### ConcurrentHashMap（6题）

**Q0: HashMap和ConcurrentHashMap有什么区别？**

**标准答案**：
两者底层都基于哈希表，JDK 1.8 中都采用了数组+链表+红黑树结构，但核心区别在并发安全能力：
1. `HashMap` 是非线程安全的，适合单线程场景
2. `ConcurrentHashMap` 是线程安全的，适合多线程共享数据场景
3. `HashMap` 没有并发控制，多线程下可能导致数据覆盖、数据丢失，JDK 1.7 扩容时甚至可能形成环
4. `ConcurrentHashMap` 在 JDK 1.8 中通过 `volatile + CAS + synchronized` 保证并发安全，读操作大多无锁，写操作是桶级别的细粒度同步
5. `HashMap` 允许一个 `null key` 和多个 `null value`，`ConcurrentHashMap` 的 `key` 和 `value` 都不允许为 `null`

面试表达时要点：
`HashMap` 适合普通单线程容器场景，`ConcurrentHashMap` 适合本地缓存、共享配置、计数器这类并发读写场景。它不是简单地给整个 Map 加一把大锁，而是尽量通过无锁读和细粒度同步提升并发性能。

**追问**：
- 为什么 `ConcurrentHashMap` 不允许 `null`？
- 为什么说 `ConcurrentHashMap` 只保证单次操作线程安全，不保证复合操作原子性？

---

**Q1: ConcurrentHashMap如何保证线程安全？**

**标准答案**：
- JDK 1.7：使用Segment分段锁，将数据分成多个Segment，不同Segment可以并发访问
- JDK 1.8：取消Segment，使用CAS+Synchronized，锁粒度更细（锁定数组元素），并发性能更高

如果再展开一点，JDK 1.8 的核心思路其实是：
1. 读操作尽量无锁，通过 `volatile` 保证可见性
2. 写操作先尝试用 CAS 完成
3. CAS 失败或者发生桶冲突时，再对桶节点加 `synchronized`
4. 扩容时允许多个线程协助迁移数据

所以它不是简单地锁整张表，而是“无锁读 + 细粒度写锁 + CAS”的组合。这也是它比 `Hashtable` 并发性能更好的关键原因。

**追问**：
- Segment的数量可以动态调整吗？
- 为什么1.8不继续使用ReentrantLock？

---

**Q2: JDK 1.7和1.8的ConcurrentHashMap有什么区别？**

**标准答案**：
1. 锁机制：1.7是Segment分段锁，1.8是CAS+Synchronized
2. 锁粒度：1.7锁定Segment，1.8锁定数组元素
3. 并发度：1.7最大并发度=Segment数量，1.8=数组长度
4. 数据结构：1.8增加了红黑树

如果面试官继续追问，这题真正想听的是“为什么 1.8 改掉了 Segment”。

原因主要有三点：
1. `Segment` 结构让实现更复杂
2. 并发粒度仍然不够细，热点段竞争明显
3. JDK 1.8 想让整体结构更接近 `HashMap`，便于统一维护和优化

所以 JDK 1.8 不是简单换了锁，而是整体改成了更轻量的并发控制模型：结构像 `HashMap`，并发控制依赖 `CAS + synchronized + volatile`。

**追问**：
- 1.8的并发性能提升了多少？
- 为什么1.8选择Synchronized而不是ReentrantLock？

**Q3: ConcurrentHashMap的get操作需要加锁吗？**

**标准答案**：
不需要。`get` 操作大多数情况下是无锁的。

原因是：
1. `ConcurrentHashMap` 的关键节点和数组引用使用了 `volatile`
2. `volatile` 可以保证可见性，读线程能够看到其他线程已经发布的最新值
3. `get` 本身只是读取桶和节点引用，不需要像写操作那样修改结构

所以即使其他线程正在并发写入，`get` 也不会读到“半初始化”的中间状态，最多读到旧值或者最新值。

面试里这题的重点不是只说“不加锁”，而是要补一句：
它之所以能无锁读，是因为底层发布和读取过程配合了 `volatile` 可见性语义。

**追问**：
- volatile如何保证可见性？
- 为什么get不需要加锁也能保证线程安全？

**Q4: ConcurrentHashMap的size()方法如何实现？**

**标准答案**：
- JDK 1.7：遍历所有Segment，累加count。如果两次遍历结果一致，返回；否则加锁重新计算
- JDK 1.8：使用baseCount+counterCells数组，通过CAS更新，避免加锁

为什么不直接给整个表加锁再统计？
因为 `ConcurrentHashMap` 的设计目标就是高并发，如果每次 `size()` 都锁整张表，代价太大，会严重影响吞吐。

所以它采用的是一种“分散计数”的思路：
1. 常规情况下先更新 `baseCount`
2. 高并发冲突时，再把计数压力分摊到 `counterCells`
3. 最终统计时把这些值加总起来

因此 `size()` 更像是为了兼顾性能做的一种近似实时统计实现，而不是每次都以全局大锁换强一致。

**追问**：
- 为什么不直接加锁计算size？
- counterCells数组的作用是什么？

**Q5: ConcurrentHashMap能完全替代Hashtable吗？**

**标准答案**：
大部分场景可以替代，但有区别：
- ConcurrentHashMap不允许key和value为null
- Hashtable的迭代器是强一致性，ConcurrentHashMap是弱一致性
- 如果需要强一致性，可以使用`Collections.synchronizedMap()`

从工程角度看，绝大多数新项目里确实更推荐 `ConcurrentHashMap`，因为：
1. 并发性能更好
2. 锁粒度更细
3. 设计更现代

但不能机械地说“完全替代”，因为它们的语义并不完全一致。尤其是：
1. `null` 支持不同
2. 迭代一致性不同
3. 某些严格同步语义场景，`ConcurrentHashMap` 并不等于“所有组合操作都安全”

所以更准确的面试表达是：
大多数并发容器场景下优先用 `ConcurrentHashMap`，但要注意它和 `Hashtable` 在语义上并不是 100% 等价替换。

**追问**：
- 为什么ConcurrentHashMap不允许null？
- 什么是弱一致性？

### JUC并发（10题）

**Q1: Synchronized的锁升级过程？**

**标准答案**：
无锁→偏向锁→轻量级锁→重量级锁（只能升级，不能降级）
- 偏向锁：只有一个线程访问，在对象头记录线程ID
- 轻量级锁：多个线程交替访问，通过CAS获取锁
- 重量级锁：多个线程竞争，基于操作系统互斥量

这套升级机制的核心目标是：
不要一上来就把锁做得很重，而是根据竞争程度逐步提高成本。

可以这样理解：
1. 如果基本没有竞争，就用偏向锁，连 CAS 都尽量省掉
2. 如果有少量线程交替竞争，就升级为轻量级锁，用 CAS 自旋
3. 如果竞争激烈，自旋已经不划算了，就升级为重量级锁，线程进入阻塞和唤醒

所以锁升级本质上是 JVM 在做性能权衡：
低竞争时追求更低开销，高竞争时追求更稳定的同步语义。

**追问**：
- 为什么锁不能降级？
- 偏向锁的撤销过程？

**Q2: Volatile的作用？**

**标准答案**：
1. 可见性：一个线程修改后，其他线程立即可见
2. 禁止指令重排序：通过内存屏障实现

注意：volatile不保证原子性，`i++`操作不是线程安全的。

为什么它不能保证原子性？
因为像 `i++` 这种操作本质上包含三步：
1. 读
2. 改
3. 写

`volatile` 只能保证你读到的是最新值、写出去后别人能看到，但不能把这三步合并成一个不可分割的原子动作。

所以它适合的场景通常是：
1. 状态标记位，比如停止线程的 `running`
2. 单次写、多次读的配置刷新
3. 配合双重检查锁中的 `volatile` 防止指令重排

不适合直接拿来做计数器累加这类复合操作。

**追问**：
- 内存屏障有哪几种？
- 为什么volatile不保证原子性？

**Q3: 线程池的7个参数？**

**标准答案**：
1. corePoolSize：核心线程数
2. maximumPoolSize：最大线程数
3. keepAliveTime：空闲线程存活时间
4. unit：时间单位
5. workQueue：任务队列
6. threadFactory：线程工厂
7. handler：拒绝策略

这 7 个参数不要只死记，面试里最好讲出它们分别控制什么：
1. `corePoolSize` 决定常驻工作线程规模
2. `maximumPoolSize` 决定高峰期最多能扩到多少线程
3. `keepAliveTime + unit` 决定非核心线程空闲多久回收
4. `workQueue` 决定任务是先排队还是先扩线程
5. `threadFactory` 决定线程怎么创建，比如线程名、是否守护线程
6. `handler` 决定线程和队列都满了之后如何处理新任务

所以线程池参数设计本质上是在平衡：
并发能力、资源占用、任务等待时间和系统稳定性。

**追问**：
- 核心线程会被回收吗？
- 如何选择合适的线程池参数？

**Q4: 线程池的执行流程？**

**标准答案**：
1. 线程数<corePoolSize → 创建核心线程
2. 线程数=corePoolSize → 任务放入队列
3. 队列满 && 线程数<maximumPoolSize → 创建非核心线程
4. 队列满 && 线程数=maximumPoolSize → 执行拒绝策略

这个流程反映了线程池的设计思想：
优先控制线程数量，再利用队列做缓冲，最后才在高压场景下扩容线程数。

为什么不是先一路扩到最大线程数？
因为线程本身也是资源，线程太多会带来：
1. 上下文切换成本
2. 内存开销
3. 调度压力

所以线程池默认是“先核心线程，再队列，再非核心线程，最后拒绝”，这是一种更稳妥的资源控制策略。

**追问**：
- 为什么不是先创建到最大线程数再放队列？
- 如何动态调整线程池参数？

**Q5: 4种拒绝策略？**

**标准答案**：
1. AbortPolicy（默认）：抛出异常
2. CallerRunsPolicy：调用者线程执行
3. DiscardPolicy：直接丢弃
4. DiscardOldestPolicy：丢弃最老的任务

面试里这题最好再补一句使用倾向：
1. `AbortPolicy` 最容易暴露问题，适合希望尽早失败并监控告警的场景
2. `CallerRunsPolicy` 可以做一种天然限流，让提交方自己承担压力
3. `DiscardPolicy` 和 `DiscardOldestPolicy` 风险更大，适合允许丢任务的非核心业务

所以拒绝策略不是背名字，而是要根据业务是否允许失败、允许阻塞、允许丢任务来选。

**追问**：
- 如何自定义拒绝策略？
- 哪种拒绝策略最常用？

**Q6: AQS的核心原理？**

**标准答案**：
AQS 的核心可以概括成两部分：
1. 一个 `state` 状态变量
2. 一个双向等待队列（常被称为 CLH 变体队列）

工作流程是：
1. 线程先尝试通过 CAS 修改 `state` 获取同步状态
2. 如果获取失败，就把当前线程包装成节点加入等待队列
3. 队列中的线程按顺序阻塞等待
4. 当前持有锁的线程释放锁后，会唤醒队列中的后继节点继续竞争

所以 AQS 本质上是：
用 `state` 表示资源状态，用队列管理竞争失败的线程。

像 `ReentrantLock`、`Semaphore`、`CountDownLatch` 这类并发工具，底层很多都是基于 AQS 做状态管理和线程排队的。

- 公平锁和非公平锁的区别？

**Q7: ReentrantLock和Synchronized的区别？**

**标准答案**：
1. 实现：ReentrantLock是API层面，Synchronized是JVM层面
2. 灵活性：ReentrantLock支持公平锁、可中断、超时
3. 性能：JDK 1.6后Synchronized优化，性能相当
4. 使用：ReentrantLock需要手动释放锁

如果从“什么时候选谁”来讲，更实用：
1. 如果只是普通互斥同步，`synchronized` 更简单，语义清晰，不容易忘记释放锁
2. 如果你需要公平锁、可中断获取锁、超时获取锁、多条件队列等能力，就更适合 `ReentrantLock`

所以现在两者的核心差异已经不是“谁性能更高”，而是“谁的控制能力更强、谁的代码更简单”。

**Q8: CAS的ABA问题？**

**标准答案**：
线程1读取A，线程2将A改成B再改回A，线程1 CAS成功但实际值已经变化过。

解决方案：使用AtomicStampedReference，增加版本号，CAS时同时比较值和版本号。

ABA 问题的本质是：
CAS 只比较“当前值是不是还是原来的值”，但它不知道这个值在中间有没有被改过。

所以在一些只关心“最终值”的场景里，ABA 不一定有害；但如果中间变化本身有业务意义，比如：
1. 无锁栈
2. 链表节点复用
3. 某些状态机转换

那 ABA 就可能导致逻辑错误。

所以解决 ABA 的关键不是单纯比较值，而是额外引入版本信息。

**Q9: Condition的作用？**

**标准答案**：
Condition是AQS提供的等待/通知机制，类似Object的wait/notify。每个Condition维护一个等待队列，`await()`释放锁并加入等待队列，`signal()`将线程从等待队列移到CLH队列。

它相比 `wait/notify` 的优势在于更灵活：
1. 一个 `Lock` 可以绑定多个 `Condition`
2. 不同条件可以有各自独立的等待队列

这在生产者消费者模型里特别有用，比如：
1. 一个条件队列等“队列不空”
2. 另一个条件队列等“队列不满”

这样比所有线程都挤在同一个对象监视器的等待队列里更清晰，也更容易精确唤醒。

**Q10: ThreadLocal的原理？**

**标准答案**：
`ThreadLocal` 的核心思想不是“给变量加锁”，而是“给每个线程单独存一份副本”。

它的原理是：
1. 每个 `Thread` 对象内部都有一个 `ThreadLocalMap`
2. `ThreadLocalMap` 的 `key` 是 `ThreadLocal` 对象本身
3. `value` 是这个线程对应保存的线程私有变量

所以 `ThreadLocal` 的 `get/set` 本质上不是去操作某个共享变量，而是去操作“当前线程自己的那份 Map”。

这也是为什么它适合存：
1. 用户上下文
2. 数据库连接上下文
3. TraceId
4. 线程内共享但线程间隔离的数据

注意它的风险点：
`ThreadLocalMap` 中的 key 是弱引用，如果 `ThreadLocal` 对象被回收了，但线程还活着，value 可能残留，在线程池场景下就容易造成内存泄漏或脏数据串用，所以使用完通常要 `remove()`。

注意：使用后必须`remove()`，否则可能内存泄漏。

### 索引相关（10题）

**Q1: 聚簇索引和非聚簇索引的区别？**

**标准答案**：
1. 存储内容：聚簇索引叶子节点存完整行数据，非聚簇索引存索引列值+主键值
2. 数量限制：聚簇索引一张表只能有一个，非聚簇索引可以有多个
3. 查询性能：聚簇索引无需回表，非聚簇索引可能需要回表

更准确地说，在 InnoDB 里：
1. 主键索引通常就是聚簇索引
2. 聚簇索引的叶子节点直接存整行数据
3. 二级索引的叶子节点存的是索引列值 + 主键值

所以通过二级索引查完整记录时，往往还要再根据主键去聚簇索引中查一次，这就是回表。

面试里这题经常和“回表”“覆盖索引”连着问，所以最好把这三者放在一起理解。

**Q2: 什么是回表？如何避免？**

**标准答案**：
回表：通过非聚簇索引查询时，先找到主键值，再通过主键索引查找完整数据的过程。

如何避免：使用覆盖索引，让查询的所有字段都包含在索引中。

为什么回表会慢？
因为二级索引定位到主键之后，还要再去聚簇索引查一次整行数据，相当于多做了一次查找，尤其在扫描记录较多时，随机 I/O 成本会更明显。

如何避免回表，最常见的思路就是覆盖索引：
如果查询需要的列都已经在索引里了，就可以直接从索引叶子节点返回数据，不需要再回到主键索引取整行。

**追问**：

**Q3: 为什么 WHERE age + 1 = 26 不能使用索引？**

**标准答案**：
因为B+树索引存储的是age的原始值，而不是age+1的值。当对索引列进行计算时，MySQL无法直接在B+树中查找，必须对每一行计算age+1，然后判断是否等于26，这就变成了全表扫描。

正确写法：`WHERE age = 25`

这类问题本质上属于“对索引列做了表达式计算”，破坏了索引的原始有序性。

你可以这样记：
索引能生效的前提，是 MySQL 能拿着你给的条件直接去 B+ 树里定位；一旦你把列做了函数、运算、隐式转换，数据库就很难直接利用原始索引结构。

**Q4: 为什么 LIKE '%张%' 不能使用索引？**

**标准答案**：
因为B+树索引是有序的，只有前缀匹配才能利用索引的有序性快速定位。中间匹配或后缀匹配无法利用有序性，必须全表扫描。

可以使用索引的写法：`LIKE '张%'`（前缀匹配）

本质原因还是“有序性”。
B+ 树能高效查找，是因为它知道从左到右怎么排。如果写成 `%张%`，最左边已经不确定了，数据库没法利用索引从某个固定起点开始定位，只能扫描大量数据再做匹配。

**Q5: 什么是覆盖索引？有什么优势？**

**标准答案**：
覆盖索引：查询的所有字段都包含在索引中，无需回表，直接从索引获取数据。

优势：
1. 减少磁盘I/O（无需回表）
2. 查询性能提升约50%
3. 减少随机I/O，提高缓存命中率

面试里更稳妥的表达是：
覆盖索引最大的价值不是一个固定的百分比提升，而是避免回表。

所以什么时候特别有价值？
1. 查询记录数较多
2. 二级索引命中很多行
3. 但最终只需要少量列返回

这时候如果能用覆盖索引，收益通常很明显。

**Q7: 索引下推（ICP）是什么？**

**标准答案**：
索引下推（Index Condition Pushdown）：MySQL 5.6引入的优化，将部分WHERE条件下推到存储引擎层，减少回表次数。

例如：`WHERE name LIKE '张%' AND age=25`
- 没有ICP：先通过name索引找到所有'张%'的记录，回表后再过滤age=25
- 有ICP：在索引层就过滤age=25，减少回表次数

它的核心价值是：
不是改变索引结构，而是把原本要到 Server 层再判断的条件，尽量提前到存储引擎层处理。

这样做的好处就是：
能在回表之前先过滤掉一部分不满足条件的记录，从而减少不必要的主键回查。

**Q8: B+树和B树的区别？**

**标准答案**：
1. 数据存储：B树所有节点都存数据，B+树只有叶子节点存数据
2. 节点容量：B+树非叶子节点只存key，能存更多索引，树更矮
3. 范围查询：B+树叶子节点形成链表，范围查询直接遍历链表，B树需要中序遍历

为什么数据库更喜欢 B+ 树？
因为数据库场景特别在意磁盘 I/O 和范围查询效率。

B+ 树的优势在于：
1. 非叶子节点不存行数据，只存 key，所以一个页里能放更多索引项，树高度更低
2. 所有数据都在叶子节点，查询路径更稳定
3. 叶子节点天然有序且形成链表，范围查询和排序非常方便

所以数据库索引更关心“稳定的磁盘访问次数”和“范围扫描能力”，这正是 B+ 树比 B 树更适合的地方。

**Q9: 为什么MySQL使用B+树而不是哈希表？**

**标准答案**：
哈希表的优势是精确匹配很快，但它只适合 `=` 这类查询。

数据库真实场景里，经常需要：
1. 范围查询，比如 `> < between`
2. 排序，比如 `order by`
3. 前缀匹配
4. 最左前缀匹配的联合索引能力

这些都依赖“有序性”，而哈希表天然无序，所以很难支持。

B+ 树则不同：
1. 它是有序结构
2. 支持范围查询
3. 支持排序
4. 支持联合索引的最左前缀匹配

所以 MySQL 选 B+ 树不是因为哈希不快，而是因为数据库需要的不只是“等值快查”，还要兼顾范围、排序和磁盘访问效率。

### MVCC相关（5题）

**Q1: 什么是MVCC？它解决了什么问题？**

**标准答案**：
MVCC，全称多版本并发控制，是 InnoDB 在并发读场景下实现事务隔离的重要机制。

它的核心思想不是“所有读都加锁”，而是：
为一行数据维护多个历史版本，不同事务根据自己的快照去读不同版本的数据。

它主要解决的问题是：
1. 在并发场景下，普通读操作不想频繁加锁
2. 又希望事务内读到一致的数据结果

所以 InnoDB 通过：
1. `Undo Log` 保存历史版本
2. `ReadView` 决定当前事务能看到哪个版本

来实现一致性读。

你可以把 MVCC 理解成：
“同一行数据在逻辑上有多个版本，不同事务各自看自己应该看到的那个版本。”

**追问**：
- MVCC能解决幻读吗？
- MVCC和锁的关系？

---

**Q2: MVCC是如何实现可重复读的？**

**标准答案**：
1. 版本链：通过Undo Log保存数据的历史版本
2. ReadView：事务第一次查询时创建快照，记录当前活跃事务列表
3. 可见性判断：根据ReadView判断版本链上哪些版本可见
4. 复用ReadView：事务内所有查询都使用同一个ReadView，保证数据一致性

如果展开讲，它的关键链路是这样的：
1. 一行数据被修改时，旧版本不会立刻消失，而是通过 `Undo Log` 串成版本链
2. 事务第一次做快照读时，会生成一个 `ReadView`
3. `ReadView` 会记录当前哪些事务还活跃，哪些事务已经提交
4. 查询时，InnoDB 会沿着版本链往回找，直到找到一个对当前事务“可见”的版本
5. 在 `RR` 隔离级别下，同一个事务内会复用第一次生成的 `ReadView`，所以多次查询看到的结果保持一致

这就是可重复读的关键：
不是数据没变，而是你始终按照同一个“快照视角”去看数据。

**追问**：
- ReadView什么时候创建？
- 为什么要复用ReadView？

---

**Q3: ReadView包含哪些信息？**

**标准答案**：
- m_ids：当前活跃的事务ID列表（未提交的事务）
- min_trx_id：最小的活跃事务ID
- max_trx_id：下一个要分配的事务ID
- creator_trx_id：创建这个ReadView的事务ID

这些字段不是为了背名词，而是为了做“可见性判断”。

你可以这样理解：
1. `m_ids` 告诉当前还有哪些事务没提交
2. `min_trx_id` 表示当前活跃事务里最小的事务 ID
3. `max_trx_id` 表示之后新事务的起点
4. `creator_trx_id` 表示这个快照是谁创建的

InnoDB 会拿数据版本上的事务 ID 去和这些信息比较，从而判断：
1. 这个版本是不是自己改的
2. 是不是别的已提交事务生成的
3. 还是仍然属于未提交事务生成的版本

所以 ReadView 本质上就是：
“当前事务看世界的一张可见性快照规则表”。

**追问**：
- 为什么需要这些信息？
- 如何根据ReadView判断可见性？

---

**Q4: RC和RR隔离级别在MVCC实现上有什么区别？**

**标准答案**：
- RC（读已提交）：每次查询都创建新的ReadView，可以读到其他事务已提交的修改
- RR（可重复读）：第一次查询时创建ReadView，后续查询复用，保证事务内多次查询结果一致

这题的关键不是只背一句“一个每次创建，一个复用”，而是理解这会带来什么结果。

在 `RC` 下：
1. 你每查一次，都会站在一个新的时间点上看数据
2. 所以别的事务只要已经提交，你下一次查询就可能看到新结果
3. 因此会出现不可重复读

在 `RR` 下：
1. 第一次快照读时确定视角
2. 后续快照读都沿用这个视角
3. 所以事务内多次读取结果保持一致

所以可以这么记：
`RC` 更强调“读到别人最新已提交结果”，`RR` 更强调“同一事务内读取一致性”。

**追问**：
- 为什么RC每次都创建ReadView？
- RR如何解决不可重复读？

**Q5: MVCC能完全避免锁吗？**

**标准答案**：
不能。MVCC只解决读-写冲突，不解决写-写冲突。写操作仍然需要加锁（行锁、Gap锁等）。

也就是说：
1. 普通 `select` 这类快照读，很多时候可以靠 MVCC 不加锁完成
2. 但 `update`、`delete`、`select ... for update` 这种当前读，仍然需要锁

所以不要把 MVCC 理解成“数据库从此不需要锁了”。更准确地说，InnoDB 是通过 **MVCC + 锁机制** 一起实现隔离级别和并发控制的。

- 什么是写-写冲突？

### 事务隔离级别相关（5题）

**Q1: MySQL的四种事务隔离级别是什么？各有什么特点？**

**标准答案**：
从低到高：
1. **READ UNCOMMITTED（读未提交）**：可能出现脏读、不可重复读、幻读
2. **READ COMMITTED（读已提交）**：避免脏读，可能出现不可重复读、幻读（Oracle默认）
3. **REPEATABLE READ（可重复读）**：避免脏读和不可重复读，可能出现幻读（MySQL默认）
4. **SERIALIZABLE（串行化）**：避免所有并发问题，但性能最差

本质上可以这样理解：
1. 隔离级别越低，并发能力通常越强，但数据问题越多
2. 隔离级别越高，一致性越强，但锁冲突和等待通常越多

所以事务隔离级别从来不是“越高越先进”，而是看业务在一致性和性能之间怎么权衡。

- 如何选择合适的隔离级别？

**Q2: 什么是脏读、不可重复读、幻读？**

**标准答案**：
- **脏读**：读到其他事务**未提交**的数据。事务A读到事务B未提交的修改，但事务B最终回滚了
- **不可重复读**：同一事务内两次读取**同一行数据**结果不同。事务A两次读取，中间事务B修改并提交了
- **幻读**：同一事务内两次查询**记录数**不同。事务A两次查询，中间事务B插入或删除了记录

区分这三个概念时，最容易混的是“不可重复读”和“幻读”。

可以这样记：
1. 脏读关注的是“读到了未提交数据”
2. 不可重复读关注的是“同一行内容变了”
3. 幻读关注的是“同一范围结果集变了”

所以一个强调“行内容变化”，一个强调“结果集数量变化”。

**Q3: MySQL如何解决幻读？**

**标准答案**：
MySQL通过 **MVCC + Next-Key Lock** 组合解决幻读：

1. **快照读（普通SELECT）**：通过MVCC实现，读取历史版本，看不到新插入的记录
2. **当前读（SELECT FOR UPDATE、UPDATE、DELETE）**：通过Next-Key Lock（记录锁+间隙锁）锁定范围，阻止其他事务插入

**示例**：
```sql
-- 事务A
BEGIN;
SELECT * FROM user WHERE age BETWEEN 20 AND 30;  -- 快照读，不加锁
-- 事务B可以插入age=25的记录

SELECT * FROM user WHERE age BETWEEN 20 AND 30 FOR UPDATE;  -- 当前读，加Next-Key Lock
-- 事务B无法插入age=25的记录（被阻塞）
```

这题真正要讲清的是：
MySQL 不是靠单一手段解决幻读，而是分场景处理。

1. 对于普通 `select` 这种快照读，靠 MVCC 让事务始终读自己的快照，因此很多“新插入的记录”对当前事务不可见
2. 对于 `select for update`、`update`、`delete` 这种当前读，靠 Next-Key Lock 锁住记录和间隙，阻止别人往这个范围里插数据

所以面试里更准确的表述是：
InnoDB 通过 **MVCC 处理快照读的一致性问题，通过 Next-Key Lock 处理当前读下的插入并发问题**，两者配合控制幻读。

**Q4: 为什么MySQL默认隔离级别是RR而不是RC？**

**标准答案**：
主要原因是**主从复制的一致性**：

1. **MySQL 5.0之前**：binlog只支持Statement格式（记录SQL语句）
2. **RC级别问题**：在RC级别下，同一条SQL在主库和从库执行可能产生不同结果（因为可以读到其他事务的提交）
3. **RR级别优势**：保证事务内读取一致，主从复制更安全

**现在**：MySQL 5.1后支持Row格式的binlog，RC级别也可以安全使用，但为了兼容性，默认仍是RR。

如果从今天的工程视角再补一句：
现在很多系统在特定场景下也会主动选择 `RC`，比如更关注减少锁范围、降低间隙锁影响的场景。

所以这题不要答成“RR 永远比 RC 更好”，更准确的说法是：
MySQL 默认选择 RR 有历史和兼容性原因，但具体项目仍然要按业务场景选。

**Q5: 当前读和快照读的区别？**

**标准答案**：

| 对比 | 快照读 | 当前读 |
|------|--------|--------|
| **定义** | 读取历史版本（MVCC） | 读取最新版本（加锁） |
| **SQL** | 普通SELECT | SELECT FOR UPDATE、UPDATE、DELETE、INSERT |
| **是否加锁** | 不加锁 | 加锁（共享锁或排他锁） |
| **读取数据** | ReadView确定的版本 | 当前最新提交的版本 |
| **并发性** | 高（不阻塞） | 低（可能阻塞） |

**示例**：
```sql
-- 快照读
SELECT * FROM user WHERE id=1;  -- 读取ReadView确定的版本，不加锁

-- 当前读
SELECT * FROM user WHERE id=1 FOR UPDATE;  -- 读取最新版本，加排他锁
SELECT * FROM user WHERE id=1 LOCK IN SHARE MODE;  -- 读取最新版本，加共享锁
UPDATE user SET name='张三' WHERE id=1;  -- 当前读，加排他锁
```

可以进一步这样理解：
1. 快照读适合“我主要想看一个一致结果，不想影响别人，也不想被别人影响”
2. 当前读适合“我接下来要修改数据，必须确保我拿到的是最新值，并且把别人先挡住”

所以为什么 `update` 必须是当前读？
因为你不能拿着历史版本去改数据，必须先读到当前最新版本，再加锁修改，否则就会产生并发覆盖问题。

**追问**：

- 如何选择快照读还是当前读？

### 缓存问题（1题）

**Q1: 缓存穿透和缓存雪崩是什么？**

**标准答案**：
两者都是缓存没能有效挡住数据库压力，但原因不同：

1. **缓存穿透**
查询的数据本身就不存在，缓存里没有，数据库里也没有。如果没有做特殊处理，请求每次都会穿透缓存打到数据库。

常见解决方案：
- 缓存空值
- 布隆过滤器
- 参数合法性校验

2. **缓存雪崩**
大量缓存 key 在同一时间失效，或者 Redis 整体不可用，导致原本应该由缓存承接的请求瞬间全部打到数据库。

常见解决方案：
- 过期时间加随机值，打散失效时间
- 热点数据不过期或采用逻辑过期
- 缓存重建时加锁，避免并发回源
- 限流、降级、熔断
- Redis 主从、哨兵、集群等高可用方案

区分要点：
穿透是“查不存在的数据”，雪崩是“大面积缓存同时失效”。面试里还可以顺手补一句：击穿是“单个热点 key 失效”。

- 缓存击穿和缓存雪崩怎么区分？

**Q2: 你做过库存扣减，怎么避免超卖？**

**标准答案**：

最基础也是最常见的方案是数据库条件更新：

```sql
update product
set stock = stock - 1
where id = ? and stock > 0;
```

通过受影响行数判断是否扣减成功：
- 影响 1 行，说明成功
- 影响 0 行，说明库存不足

这个方案的核心是：
把“判断库存是否充足”和“执行扣减”合并成一条原子 SQL，避免“先查再改”带来的并发问题。

- Redis 原子预扣减
- 消息队列异步下单/落库
- 幂等控制
- 补偿机制
- 限流降级

成熟表达：
数据库原子扣减是防超卖的基础方案；高并发场景再配合 Redis 预扣、MQ 和补偿机制，兼顾性能和最终一致性。

**Q12: 二叉树前序遍历如何迭代实现？**

**标准答案**：
使用栈模拟递归。先将根节点入栈，然后不断弹出栈顶节点访问，将右左子节点入栈（顺序反是为了让左节点先被处理）。

```java
public List<Integer> preorderTraversal(TreeNode root) {
    List<Integer> result = new ArrayList<>();
    Stack<TreeNode> stack = new Stack<>();
    if (root != null) stack.push(root);
    
    while (!stack.isEmpty()) {
        TreeNode node = stack.pop();
        result.add(node.val);
        if (node.right != null) stack.push(node.right);
        if (node.left != null) stack.push(node.left);
    }
    return result;
}
```

**Q13: 二叉树中序遍历如何迭代实现？**

**标准答案**：
使用栈模拟递归。一直向左遍历入栈，然后弹出访问，再向右遍历。

```java
public List<Integer> inorderTraversal(TreeNode root) {
    List<Integer> result = new ArrayList<>();
    Stack<TreeNode> stack = new Stack<>();
    TreeNode curr = root;
    
    while (curr != null || !stack.isEmpty()) {
        while (curr != null) {
            stack.push(curr);
            curr = curr.left;
        }
        curr = stack.pop();
        result.add(curr.val);
        curr = curr.right;
    }
    return result;
}
```

**Q14: 如何实现二叉树的层序遍历？**

**标准答案**：

```java
public List<List<Integer>> levelOrder(TreeNode root) {
    List<List<Integer>> result = new ArrayList<>();
    Queue<TreeNode> queue = new LinkedList<>();
    if (root != null) queue.offer(root);
    
    while (!queue.isEmpty()) {
        int levelSize = queue.size();
        List<Integer> level = new ArrayList<>();
        for (int i = 0; i < levelSize; i++) {
            TreeNode node = queue.poll();
            level.add(node.val);
            if (node.left != null) queue.offer(node.left);
            if (node.right != null) queue.offer(node.right);
        }
        result.add(level);
    }
    return result;
}
```

**Q15: 深度优先遍历和广度优先遍历的区别？**

**标准答案**：
- DFS（深度优先）：一直往深处走，走到底再回头。用栈或递归实现，适合查看路径、拓扑排序
- BFS（广度优先）：一层一层遍历。用队列实现，适合最短路径、层序遍历

| 对比 | DFS | BFS |
|------|-----|-----|
| 实现方式 | 栈/递归 | 队列 |
| 空间复杂度 | O(h) | O(w) |
| 适用场景 | 路径、拓扑排序 | 最短路径、层序 |

### 1. Java 中 equals 和 == 有什么区别？

`==` 和 `equals` 的核心区别可以先记一句：
`==` 比较的是“是不是同一个东西”，`equals` 比较的是“内容上算不算相等”。

先说 `==`。在 Java 里，`==` 是运算符，它分两种情况：
1. 如果比较的是基本类型，比如 `int`、`long`，那比较的是值
2. 如果比较的是引用类型，比如 `String`、`User`，那比较的是两个引用是不是指向同一个对象，也就是地址是不是同一个

再说 `equals`。`equals` 是 `Object` 类的方法，默认实现其实和 `==` 很像，本质上也是比较地址。但很多类会重写它，比如 `String`、`Integer`，重写之后就变成了比较内容。

所以你在开发里可以这样理解：
1. 比较基本类型值，用 `==`
2. 比较两个对象的业务内容是否相等，通常用 `equals`
3. 如果你就是想判断是不是同一个对象实例，那才用 `==`

再补几个面试里容易追问的点：
1. `String` 不能用 `==` 比较内容，因为字符串常量池会让有些场景下 `==` 恰好是 `true`，但这不可靠
2. 包装类型也不要依赖 `==`，比如 `Integer` 有 `-128~127` 的缓存
3. 如果对象可能为 `null`，更稳妥的写法是 `Objects.equals(a, b)`

面试里我一般会这么答：
`==` 是运算符，基本类型比较的是值，引用类型比较的是地址；`equals` 是 `Object` 的方法，默认也是比较地址，但很多类会重写成比较内容，所以业务开发里比较对象内容一般用 `equals`，判断是否同一个对象才用 `==`。

### 2. HashMap 和 ConcurrentHashMap 有什么区别？

这题的核心不是背定义，而是先抓住“线程安全”这条主线。

`HashMap` 是非线程安全的，适合单线程场景；`ConcurrentHashMap` 是线程安全的，适合多线程并发访问场景。

它们在 JDK 1.8 里的底层结构其实很像，都是数组 + 链表 + 红黑树。区别不在结构本身，而在并发控制方式。

`HashMap` 没有任何并发保护。如果多个线程同时 `put`、扩容、迁移数据，可能会出现：
1. 数据覆盖
2. 数据丢失
3. 结构异常
4. JDK 1.7 里扩容甚至可能形成环形链表

`ConcurrentHashMap` 则是为并发场景设计的。JDK 1.7 用的是 `Segment` 分段锁，JDK 1.8 取消了 `Segment`，改成 `volatile + CAS + synchronized`。它的特点是：
1. 读操作大多数情况下不加锁，所以并发读性能高
2. 写操作不是锁整张表，而是尽量做细粒度控制，通常锁桶或者桶上的节点
3. 扩容时还支持多个线程协助迁移

还有一个常考点是 `null`：
1. `HashMap` 允许一个 `null key` 和多个 `null value`
2. `ConcurrentHashMap` 不允许 `key` 和 `value` 为 `null`

原因是并发场景下，如果 `get(key)` 返回 `null`，你没法明确区分到底是：
1. 这个 key 根本不存在
2. 还是 value 本来就是 `null`

单线程下你还可以再查一次 `containsKey`，但并发场景里两次操作之间状态可能变了，所以它直接禁止 `null`。

面试里我一般会这样说：
`HashMap` 和 `ConcurrentHashMap` 底层都是哈希表，但 `HashMap` 是非线程安全的，适合单线程；`ConcurrentHashMap` 是线程安全的，适合并发场景。JDK 1.8 中它通过 `volatile`、CAS 和桶级别同步保证安全，读大多无锁，写是细粒度加锁，所以并发性能比传统的 `Hashtable` 更好。另外 `ConcurrentHashMap` 不允许 `null`，这是为了避免并发语义歧义。

### 3. 联合索引和最左前缀原则是什么？

联合索引就是把多个字段按顺序组成一个索引，比如：

```sql
index(a, b, c)
```

它不是三个独立索引，而是一个有顺序的组合索引。这个顺序非常重要，因为底层 B+ 树是先按 `a` 排，再按 `b` 排，最后按 `c` 排。

最左前缀原则的意思就是：
使用这个联合索引时，要尽量从最左边的列开始连续匹配。

比如 `(a, b, c)` 这个索引，下面这些通常都能利用：
1. `where a = 1`
2. `where a = 1 and b = 2`
3. `where a = 1 and b = 2 and c = 3`

但如果你跳过最左边，比如：
1. `where b = 2`
2. `where c = 3`
3. `where b = 2 and c = 3`

通常就没法很好地利用这个索引。

为什么会这样？本质原因不是 MySQL 人为规定，而是 B+ 树的有序性决定的。你可以把 `(a, b, c)` 理解成一本先按姓、再按名、再按年龄排好的名单。你如果连“姓”都不知道，直接按“名”查，就没法快速定位。

还有一个高频点是“连续”：
`where a = 1 and c = 3` 这种，中间跳了 `b`，通常只能稳定利用到 `a`，后面的 `c` 很难像完整命中联合索引那样高效。

再补一个常考点：范围查询。
比如：

```sql
where a = 1 and b > 10 and c = 3
```

通常 `a` 和 `b` 还能用上，但遇到范围条件后，后面的 `c` 是否还能继续高效利用，就会明显受限。

面试里我一般会这么答：
联合索引就是多个列按顺序组成的一个索引，比如 `(a, b, c)`。因为底层 B+ 树也是按这个顺序组织的，所以查询时要尽量从最左列开始连续匹配，这就是最左前缀原则。也就是说，这个索引天然支持 `(a)`、`(a,b)`、`(a,b,c)`，但不等价于 `(b)` 或 `(c)`。如果中间断列，或者前面出现范围查询，后续列的利用通常会受影响。

### 4. Explain 一般看哪些字段？分别代表什么？

看 `EXPLAIN` 时，我不会平均看所有字段，而是优先盯几个最关键的。

第一是 `key`。  
它表示“实际用了哪个索引”。如果 `key = NULL`，通常就说明没有真正用上索引，这是首先要警惕的点。

第二是 `type`。  
它表示访问类型，也就是 MySQL 怎么找数据。常见从好到差可以粗略记成：
`const`、`eq_ref`、`ref`、`range`、`index`、`ALL`
其中 `ALL` 代表全表扫描，一般最需要警惕；`ref`、`range` 往往还算可以接受。

第三是 `rows`。  
它表示 MySQL 预估要扫描多少行。这个字段非常重要，因为扫描行数越大，代价通常越高。哪怕 SQL 最终只返回 1 行，但如果 `rows` 很大，说明过程可能并不高效。

第四是 `Extra`。  
这里经常能看出一些关键优化信息，比如：
1. `Using index`：说明可能走了覆盖索引
2. `Using where`：还要继续做条件过滤，这很常见
3. `Using temporary`：用了临时表，通常要警惕
4. `Using filesort`：发生额外排序，往往是优化重点

第五是 `key_len`。  
这个字段很适合判断联合索引到底用到了哪几列。你不用死记字节数，但要知道它能帮助判断索引命中深度。

还有两个辅助字段：
1. `possible_keys`：理论上可能会用到哪些索引
2. `id` 和 `select_type`：当 SQL 有子查询、派生表、`union` 时，用来分析执行层次

所以我实际排查慢 SQL 时，顺序通常是：
1. 先看 `key` 有没有用到索引
2. 再看 `type` 是不是太差
3. 再看 `rows` 扫了多少行
4. 最后看 `Extra` 有没有临时表和额外排序

面试里我一般会这么答：
我看 `EXPLAIN` 时优先关注 `key`、`type`、`rows` 和 `Extra`。`key` 看实际用没用上索引，`type` 看访问效率，`rows` 看预计扫描行数，`Extra` 看有没有覆盖索引、临时表和额外排序。如果是联合索引场景，我还会看 `key_len` 判断索引到底用到了哪几列；如果 SQL 比较复杂，再结合 `id` 和 `select_type` 分析执行层次。

### 5. 事务隔离级别有哪些？

事务隔离级别，本质是在回答一个问题：
多个事务并发执行时，一个事务到底能在多大程度上看到另一个事务的修改。

标准里常见有四种：
1. `Read Uncommitted`，读未提交
2. `Read Committed`，读已提交
3. `Repeatable Read`，可重复读
4. `Serializable`，串行化

你要先知道它们主要是在防三类并发问题：
1. 脏读：读到了别的事务还没提交的数据
2. 不可重复读：同一个事务里，两次读同一行，结果不一样
3. 幻读：同一个事务里，两次按条件查，发现记录数变了

然后分别理解四个级别：

`Read Uncommitted` 隔离性最低。  
一个事务可以看到别的事务未提交的数据，所以脏读、不可重复读、幻读都可能发生。

`Read Committed` 的意思是只能读到别的事务已经提交的数据。  
这样解决了脏读，但仍然可能有不可重复读和幻读。Oracle 默认一般就是这个级别。

`Repeatable Read` 的意思是同一个事务里，多次读取同一条记录应该保持一致。  
它解决了脏读和不可重复读。按标准定义下仍然可能有幻读，但 MySQL InnoDB 的实现比标准更强，它通过 MVCC 和锁机制，对幻读也做了较强控制。MySQL 默认就是这个级别。

`Serializable` 隔离性最高。  
它基本上让事务像串行一样执行，所以脏读、不可重复读、幻读都能避免，但并发性能也最差。

所以本质上，隔离级别是在一致性和性能之间做权衡：
1. 隔离级别越低，并发性能通常越好，但数据问题越多
2. 隔离级别越高，一致性越强，但吞吐通常越差

面试里我一般这样答：
事务隔离级别有四种，分别是读未提交、读已提交、可重复读和串行化。读未提交可能出现脏读、不可重复读和幻读；读已提交解决脏读，但还可能有不可重复读和幻读；可重复读解决脏读和不可重复读，标准上仍可能有幻读，不过 MySQL InnoDB 通过 MVCC 和锁机制对幻读也做了较强控制；串行化隔离性最高，但并发性能最差。它本质上是在一致性和并发性能之间做取舍。

### 6. Token 登录是怎么做的？和 Session 有什么区别？

如果面试官问 Token 登录，你最好按“登录流程”来讲，而不是只说一句“返回 JWT”。

一个常见流程是这样的：
1. 用户提交用户名和密码
2. 服务端校验账号密码是否正确
3. 校验通过后，服务端生成一个 Token 返回给前端
4. 前端后续请求把 Token 放到请求头中，比如 `Authorization`
5. 服务端在过滤器、拦截器或者网关层统一校验 Token 是否合法、是否过期
6. 校验通过后解析出用户身份，放到上下文中，供业务代码使用

如果项目更偏企业级实践，通常不会只做一个纯前端保存的字符串，而是会配合 Redis 保存登录态，这样能更方便地做：
1. 过期控制
2. 续期
3. 退出登录
4. 强制下线

Session 的思路则不一样。  
Session 是用户登录成功后，服务端创建一份会话数据，并生成一个 `SessionId` 返回给客户端。客户端后续请求自动带上这个 `SessionId`，服务端再根据它找到对应会话。

所以两者最核心的区别是：
1. `Session` 是服务端保存会话状态，客户端通常只保存 `SessionId`
2. `Token` 是客户端每次显式携带令牌，服务端负责校验令牌

再从架构上看：
1. `Session` 更适合传统单体 Web 应用
2. `Token` 更适合前后端分离、移动端、微服务、网关场景

再补一个容易说错的点：
不要把 Token 说成“天然无状态”。如果你是“Token + Redis”方案，那服务端其实仍然保存了登录态，只不过这种模式更适合分布式。

面试里我一般会这样答：
我们项目里的登录认证是基于 Token 做的。用户登录时先校验账号密码，校验通过后由服务端生成 Token 返回给前端。前端后续请求把 Token 放到请求头里，服务端在过滤器、拦截器或者网关层统一校验 Token 的合法性和有效期，并解析出用户身份供业务使用。如果项目需要更强的可控性，通常会结合 Redis 保存登录态，用来做续期、登出和强制下线。和 Session 相比，Session 是服务端保存会话，客户端只带 SessionId；Token 是客户端每次显式携带令牌，更适合前后端分离和分布式场景。

### 7. 做过库存扣减，怎么避免超卖？

超卖的本质是高并发下，多个请求同时判断“还有库存”，然后都去扣减，最后实际卖出的数量超过真实库存。

最容易出问题的写法就是“先查再改”：
1. 先 `select stock`
2. 再 `update stock = stock - 1`

因为这两步之间有并发窗口。比如库存只剩 1，两个线程都查到 1，然后都扣减，就可能超卖。

所以更稳妥的做法是直接用数据库条件更新，把“判断库存”和“扣减库存”合成一条原子 SQL：

```sql
update product
set stock = stock - 1
where id = ? and stock > 0;
```

然后根据受影响行数判断结果：
1. 影响 1 行，说明扣减成功
2. 影响 0 行，说明库存不足

这就是最基础、最常见、最适合面试先讲的方案。

如果面试官继续追问更高并发场景，比如秒杀，你再往下讲：
1. 先在 Redis 做原子预扣减
2. 扣减成功后再异步创建订单、落库
3. 配合 MQ、幂等、补偿机制保证最终一致性
4. 再加上限流、降级，避免流量把数据库打爆

这里有个表达要成熟一点：
分布式锁不是不能用，但通常不是库存扣减的首选核心方案。因为热点库存场景下，分布式锁性能和复杂度都不一定最优。更常见的是数据库原子更新，或者 Redis 原子预扣减加异步落库。

面试里我一般会这样答：
库存超卖的根本原因是高并发下多个请求同时判断库存充足，然后都去扣减。我通常不会采用先查库存再更新的两步操作，而是直接使用数据库条件更新，比如 `update ... set stock = stock - 1 where id = ? and stock > 0`，把库存校验和扣减合并成一条原子 SQL，再通过受影响行数判断是否成功。如果是更高并发的秒杀场景，我会进一步用 Redis 做原子预扣减，再结合 MQ、幂等和补偿机制保证最终一致性。

### 8. 缓存穿透和缓存雪崩是什么？

这两个问题本质上都是：缓存没把请求压力挡住，流量最后打到了数据库。  
但它们的原因不同。

缓存穿透指的是：
请求访问的数据本来就不存在。缓存里没有，数据库里也没有。如果系统没有对这种“空结果”做处理，请求每次都会穿透缓存直接打到数据库。

常见场景有：
1. 恶意请求，故意查不存在的 id
2. 业务参数错误，不断查无效数据
3. 查询不到结果后，没有缓存空值

解决方案通常有三类：
1. 缓存空值
2. 布隆过滤器
3. 参数合法性校验

缓存雪崩则是另一种情况：
大量原本应该命中的缓存，在同一时间一起失效了，或者 Redis 整体不可用，结果大量请求一下子全部打到数据库。

典型场景有：
1. 批量缓存设置了相同 TTL，同一时刻一起过期
2. Redis 宕机
3. 大促时热点 key 大面积失效

解决方案通常是：
1. 给过期时间加随机值，打散失效时间
2. 热点数据不过期，或者采用逻辑过期
3. 缓存重建时加锁，避免并发回源
4. 限流、降级、熔断
5. Redis 做高可用，比如主从、哨兵、集群

面试里你最好顺手再补一句和“击穿”的区别：
1. 穿透：查不存在的数据
2. 击穿：单个热点 key 失效
3. 雪崩：大量 key 同时失效

面试里我一般会这样答：
缓存穿透是指请求的数据本身就不存在，缓存和数据库都没有，如果没有特殊处理，请求每次都会打到数据库；常见解决方案是缓存空值、布隆过滤器和参数校验。缓存雪崩是指大量缓存 key 同时失效，或者缓存系统整体不可用，导致原本应该由缓存承接的流量瞬间全部打到数据库；常见解决方案是过期时间打散、热点数据保护、缓存重建加锁、限流降级以及 Redis 高可用。

### 9. Spring Boot 和 Spring 有什么区别？

这题的关键不是说“哪个更高级”，而是讲清楚它们的关系。

`Spring` 是一个企业级基础框架，核心提供 IOC、AOP、事务管理、Web、数据访问这些底层能力。你可以把它理解成一个大的基础设施框架生态。

`Spring Boot` 不是替代 Spring，而是建立在 Spring 之上的快速开发框架。它的目标不是发明一套全新的编程模型，而是让你更容易把 Spring 用起来。

两者的主要区别可以从几个角度说：

第一，配置方式不同。  
传统 Spring 项目配置比较多，早期经常是 XML，后面也可能有大量 Java Config。  
Spring Boot 强调约定优于配置，通过自动配置和外部化配置减少手工装配成本。

第二，依赖管理方式不同。  
Spring Boot 提供了 starter 机制，比如 `spring-boot-starter-web`，把一类场景需要的依赖和兼容版本先帮你整合好。

第三，启动方式不同。  
传统 Spring Web 项目很多时候要部署到外部 Tomcat；Spring Boot 常见是内嵌 Tomcat，直接 `main` 方法启动，适合微服务和前后端分离项目。

第四，开发体验不同。  
Spring 更像是给你一堆基础零件和装配能力；Spring Boot 更像是先帮你把大部分常用零件装好，让你快速起项目。

但一定要强调一个点：
Spring Boot 的底层核心能力仍然是 Spring 提供的，比如 IOC、AOP、事务这些都没有变。所以会写 Spring Boot 不等于真正懂 Spring，面试官往往还会继续追底层原理。

面试里我一般会这样答：
Spring 是企业级基础框架，核心提供 IOC、AOP、事务管理以及 Web、数据访问等能力；Spring Boot 是基于 Spring 的快速开发框架，它不是替代 Spring，而是通过自动配置、starter 和内嵌容器等机制，简化了 Spring 项目的搭建和运行。简单说，Spring 解决的是框架能力，Spring Boot 解决的是如何更高效地把 Spring 用起来。
