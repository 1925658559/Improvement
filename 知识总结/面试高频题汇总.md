## Java 核心（20题）

### HashMap（5题）

**Q1: HashMap的底层实现原理？**

**标准答案**：
HashMap底层是数组+链表+红黑树（JDK 1.8）。通过hash函数计算key的哈希值，然后通过`(n-1) & hash`计算数组下标。如果发生哈希冲突，使用链表存储；当链表长度≥8且数组长度≥64时，转换成红黑树。

**追问**：
- 为什么是(n-1) & hash而不是hash % n？
- 为什么链表长度是8才转红黑树？

---

**Q2: JDK 1.7和1.8的HashMap有什么区别？**

**标准答案**：
1. 数据结构：1.7是数组+链表，1.8增加了红黑树
2. 插入方式：1.7是头插法，1.8是尾插法
3. 扩容优化：1.8通过`hash & oldCap`判断位置
4. 哈希算法：1.8简化了扰动函数

**追问**：
- 为什么1.7用头插法会导致死循环？
- 1.8的尾插法如何避免死循环？

---

**Q3: HashMap的扩容机制？**

**标准答案**：
当元素数量超过`容量×负载因子`（默认0.75）时触发扩容。扩容时创建新数组（容量×2），重新计算每个元素的位置。JDK 1.8优化：元素位置只有两种可能（原位置或原位置+旧容量），通过`hash & oldCap`判断。

**追问**：
- 为什么负载因子是0.75？
- 扩容时如何保证线程安全？

---

**Q4: HashMap为什么线程不安全？**

**标准答案**：
1. JDK 1.7：扩容时使用头插法，多线程可能形成环形链表，导致死循环
2. JDK 1.8：虽然改成尾插法，但put操作不是原子的，多线程可能导致数据丢失

**追问**：
- 如何让HashMap线程安全？
- ConcurrentHashMap和Hashtable的区别？

---

**Q5: 为什么链表长度达到8才转换成红黑树？**

**标准答案**：
根据泊松分布，链表长度达到8的概率极低（约0.00000006）。红黑树节点占用空间是链表的2倍，如果阈值太小会浪费空间；如果阈值太大链表查找效率低。8是性能和空间的平衡点。

**追问**：
- 红黑树什么时候退化成链表？
- 为什么退化阈值是6而不是7？

### ConcurrentHashMap（5题）

**Q1: ConcurrentHashMap如何保证线程安全？**

**标准答案**：
- JDK 1.7：使用Segment分段锁，将数据分成多个Segment，不同Segment可以并发访问
- JDK 1.8：取消Segment，使用CAS+Synchronized，锁粒度更细（锁定数组元素），并发性能更高

**追问**：
- Segment的数量可以动态调整吗？
- 为什么1.8不继续使用ReentrantLock？

---

**Q2: JDK 1.7和1.8的ConcurrentHashMap有什么区别？**

**标准答案**：
1. 锁机制：1.7是Segment分段锁，1.8是CAS+Synchronized
2. 锁粒度：1.7锁定Segment，1.8锁定数组元素
3. 并发度：1.7最大并发度=Segment数量，1.8=数组长度
4. 数据结构：1.8增加了红黑树

**追问**：
- 1.8的并发性能提升了多少？
- 为什么1.8选择Synchronized而不是ReentrantLock？

---

**Q3: ConcurrentHashMap的get操作需要加锁吗？**

**标准答案**：
不需要。get操作是无锁的，通过volatile保证可见性。即使其他线程正在修改数据，get也能读到最新值或旧值，不会读到中间状态。

**追问**：
- volatile如何保证可见性？
- 为什么get不需要加锁也能保证线程安全？

---

**Q4: ConcurrentHashMap的size()方法如何实现？**

**标准答案**：
- JDK 1.7：遍历所有Segment，累加count。如果两次遍历结果一致，返回；否则加锁重新计算
- JDK 1.8：使用baseCount+counterCells数组，通过CAS更新，避免加锁

**追问**：
- 为什么不直接加锁计算size？
- counterCells数组的作用是什么？

---

**Q5: ConcurrentHashMap能完全替代Hashtable吗？**

**标准答案**：
大部分场景可以替代，但有区别：
- ConcurrentHashMap不允许key和value为null
- Hashtable的迭代器是强一致性，ConcurrentHashMap是弱一致性
- 如果需要强一致性，可以使用`Collections.synchronizedMap()`

**追问**：
- 为什么ConcurrentHashMap不允许null？
- 什么是弱一致性？

### JUC并发（10题）

**Q1: Synchronized的锁升级过程？**

**标准答案**：
无锁→偏向锁→轻量级锁→重量级锁（只能升级，不能降级）
- 偏向锁：只有一个线程访问，在对象头记录线程ID
- 轻量级锁：多个线程交替访问，通过CAS获取锁
- 重量级锁：多个线程竞争，基于操作系统互斥量

**追问**：
- 为什么锁不能降级？
- 偏向锁的撤销过程？

---

**Q2: Volatile的作用？**

**标准答案**：
1. 可见性：一个线程修改后，其他线程立即可见
2. 禁止指令重排序：通过内存屏障实现

注意：volatile不保证原子性，`i++`操作不是线程安全的。

**追问**：
- 内存屏障有哪几种？
- 为什么volatile不保证原子性？

**Q4: 线程池的执行流程？**

**标准答案**：
1. 线程数<corePoolSize → 创建核心线程
2. 线程数=corePoolSize → 任务放入队列
3. 队列满 && 线程数<maximumPoolSize → 创建非核心线程
4. 队列满 && 线程数=maximumPoolSize → 执行拒绝策略

**追问**：

**Q6: AQS的核心原理？**

**标准答案**：
AQS维护一个state变量和一个CLH队列。线程通过CAS修改state获取锁，失败则加入CLH队列阻塞等待。释放锁时唤醒队列中的下一个线程。

**追问**：
- CLH队列是什么？
- 公平锁和非公平锁的区别？

---

**Q7: ReentrantLock和Synchronized的区别？**

**标准答案**：
1. 实现：ReentrantLock是API层面，Synchronized是JVM层面
2. 灵活性：ReentrantLock支持公平锁、可中断、超时
3. 性能：JDK 1.6后Synchronized优化，性能相当
4. 使用：ReentrantLock需要手动释放锁

**追问**：
- 什么时候用ReentrantLock？
- 如何选择锁？

---

**Q8: CAS的ABA问题？**

**标准答案**：
线程1读取A，线程2将A改成B再改回A，线程1 CAS成功但实际值已经变化过。

解决方案：使用AtomicStampedReference，增加版本号，CAS时同时比较值和版本号。

**追问**：
- ABA问题在什么场景下会出现？
- 除了版本号还有其他解决方案吗？

---

**Q9: Condition的作用？**

**标准答案**：
Condition是AQS提供的等待/通知机制，类似Object的wait/notify。每个Condition维护一个等待队列，`await()`释放锁并加入等待队列，`signal()`将线程从等待队列移到CLH队列。

**追问**：
- Condition和wait/notify的区别？
- 一个Lock可以有多个Condition吗？

---

**Q10: ThreadLocal的原理？**

**标准答案**：
每个Thread对象有一个ThreadLocalMap，key是ThreadLocal对象，value是线程私有的值。ThreadLocal的get/set操作实际是操作当前线程的ThreadLocalMap。

注意：使用后必须`remove()`，否则可能内存泄漏。

**追问**：
- 为什么会内存泄漏？
- ThreadLocal的应用场景？

## MySQL（15题）

### 索引相关（10题）

**Q1: 聚簇索引和非聚簇索引的区别？**

**标准答案**：
1. 存储内容：聚簇索引叶子节点存完整行数据，非聚簇索引存索引列值+主键值
2. 数量限制：聚簇索引一张表只能有一个，非聚簇索引可以有多个
3. 查询性能：聚簇索引无需回表，非聚簇索引可能需要回表

**追问**：
- InnoDB的主键索引是聚簇索引吗？
- 如果没有主键会怎样？

---

**Q2: 什么是回表？如何避免？**

**标准答案**：
回表：通过非聚簇索引查询时，先找到主键值，再通过主键索引查找完整数据的过程。

如何避免：使用覆盖索引，让查询的所有字段都包含在索引中。

**追问**：
- 回表的性能开销有多大？
- 什么时候应该使用覆盖索引？

---

**Q3: 为什么 WHERE age + 1 = 26 不能使用索引？**

**标准答案**：
因为B+树索引存储的是age的原始值，而不是age+1的值。当对索引列进行计算时，MySQL无法直接在B+树中查找，必须对每一行计算age+1，然后判断是否等于26，这就变成了全表扫描。

正确写法：`WHERE age = 25`

**追问**：
- 如果一定要用age+1怎么办？
- 函数索引能解决这个问题吗？

---

**Q4: 为什么 LIKE '%张%' 不能使用索引？**

**标准答案**：
因为B+树索引是有序的，只有前缀匹配才能利用索引的有序性快速定位。中间匹配或后缀匹配无法利用有序性，必须全表扫描。

可以使用索引的写法：`LIKE '张%'`（前缀匹配）

**追问**：
- 如何优化模糊查询？
- 全文索引能解决这个问题吗？

---

**Q5: 什么是覆盖索引？有什么优势？**

**标准答案**：
覆盖索引：查询的所有字段都包含在索引中，无需回表，直接从索引获取数据。

优势：
1. 减少磁盘I/O（无需回表）
2. 查询性能提升约50%
3. 减少随机I/O，提高缓存命中率

**追问**：
- 如何设计覆盖索引？
- 覆盖索引有什么缺点？

---

**Q6: 联合索引的最左前缀原则？**

**标准答案**：
联合索引遵循最左前缀原则，必须从最左边的列开始匹配。

例如索引`INDEX(a, b, c)`：
- ✅ WHERE a=1
- ✅ WHERE a=1 AND b=2
- ✅ WHERE a=1 AND b=2 AND c=3
- ❌ WHERE b=2（索引失效）

**追问**：
- 为什么要遵循最左前缀？
- WHERE a=1 AND c=3 能用到索引吗？

---

**Q7: 索引下推（ICP）是什么？**

**标准答案**：
索引下推（Index Condition Pushdown）：MySQL 5.6引入的优化，将部分WHERE条件下推到存储引擎层，减少回表次数。

例如：`WHERE name LIKE '张%' AND age=25`
- 没有ICP：先通过name索引找到所有'张%'的记录，回表后再过滤age=25
- 有ICP：在索引层就过滤age=25，减少回表次数

**追问**：
- ICP的适用场景？
- 如何判断是否使用了ICP？

---

**Q8: B+树和B树的区别？**

**标准答案**：
1. 数据存储：B树所有节点都存数据，B+树只有叶子节点存数据
2. 节点容量：B+树非叶子节点只存key，能存更多索引，树更矮
3. 范围查询：B+树叶子节点形成链表，范围查询直接遍历链表，B树需要中序遍历

**追问**：
- 为什么MySQL选择B+树而不是B树？
- B+树的高度一般是多少？

---

**Q9: 为什么MySQL使用B+树而不是哈希表？**

**标准答案**：
哈希表只支持精确查询（`=`），不支持范围查询（`BETWEEN`、`>`、`<`）和排序（`ORDER BY`）。B+树是有序的，支持范围查询和排序，更适合数据库场景。

**追问**：
- Memory引擎支持哈希索引吗？
- 什么时候用哈希索引？

---

**Q10: 索引失效的6大场景？**

**标准答案**：
1. 对索引列进行计算/函数操作
2. LIKE以通配符开头
3. 类型转换
4. 使用OR（部分字段无索引）
5. 使用!=或NOT IN
6. 联合索引不满足最左前缀

**追问**：
- 如何避免索引失效？
- 如何判断索引是否生效？

### MVCC相关（5题）

**Q1: 什么是MVCC？它解决了什么问题？**

**标准答案**：
MVCC（多版本并发控制）是InnoDB实现事务隔离级别的核心机制。通过保存数据的多个版本（Undo Log），让不同事务根据自己的ReadView看到不同的数据快照，从而实现"可重复读"隔离级别，解决"不可重复读"问题。

**追问**：
- MVCC能解决幻读吗？
- MVCC和锁的关系？

---

**Q2: MVCC是如何实现可重复读的？**

**标准答案**：
1. 版本链：通过Undo Log保存数据的历史版本
2. ReadView：事务第一次查询时创建快照，记录当前活跃事务列表
3. 可见性判断：根据ReadView判断版本链上哪些版本可见
4. 复用ReadView：事务内所有查询都使用同一个ReadView，保证数据一致性

**追问**：
- ReadView什么时候创建？
- 为什么要复用ReadView？

---

**Q3: ReadView包含哪些信息？**

**标准答案**：
- m_ids：当前活跃的事务ID列表（未提交的事务）
- min_trx_id：最小的活跃事务ID
- max_trx_id：下一个要分配的事务ID
- creator_trx_id：创建这个ReadView的事务ID

**追问**：
- 为什么需要这些信息？
- 如何根据ReadView判断可见性？

---

**Q4: RC和RR隔离级别在MVCC实现上有什么区别？**

**标准答案**：
- RC（读已提交）：每次查询都创建新的ReadView，可以读到其他事务已提交的修改
- RR（可重复读）：第一次查询时创建ReadView，后续查询复用，保证事务内多次查询结果一致

**追问**：
- 为什么RC每次都创建ReadView？
- RR如何解决不可重复读？

---

**Q5: MVCC能完全避免锁吗？**

**标准答案**：
不能。MVCC只解决读-写冲突，不解决写-写冲突。写操作仍然需要加锁（行锁、Gap锁等）。

**追问**：
- 什么是读-写冲突？
- 什么是写-写冲突？

### 事务隔离级别相关（5题）

**Q1: MySQL的四种事务隔离级别是什么？各有什么特点？**

**标准答案**：
从低到高：
1. **READ UNCOMMITTED（读未提交）**：可能出现脏读、不可重复读、幻读
2. **READ COMMITTED（读已提交）**：避免脏读，可能出现不可重复读、幻读（Oracle默认）
3. **REPEATABLE READ（可重复读）**：避免脏读和不可重复读，可能出现幻读（MySQL默认）
4. **SERIALIZABLE（串行化）**：避免所有并发问题，但性能最差

**追问**：
- 为什么MySQL默认是RR而不是RC？
- 如何选择合适的隔离级别？

---

**Q2: 什么是脏读、不可重复读、幻读？**

**标准答案**：
- **脏读**：读到其他事务**未提交**的数据。事务A读到事务B未提交的修改，但事务B最终回滚了
- **不可重复读**：同一事务内两次读取**同一行数据**结果不同。事务A两次读取，中间事务B修改并提交了
- **幻读**：同一事务内两次查询**记录数**不同。事务A两次查询，中间事务B插入或删除了记录

**追问**：
- 不可重复读和幻读的区别？
- 如何用代码演示这三种问题？

---

**Q3: MySQL如何解决幻读？**

**标准答案**：
MySQL通过 **MVCC + Next-Key Lock** 组合解决幻读：

1. **快照读（普通SELECT）**：通过MVCC实现，读取历史版本，看不到新插入的记录
2. **当前读（SELECT FOR UPDATE、UPDATE、DELETE）**：通过Next-Key Lock（记录锁+间隙锁）锁定范围，阻止其他事务插入

**示例**：
```sql
-- 事务A
BEGIN;
SELECT * FROM user WHERE age BETWEEN 20 AND 30;  -- 快照读，不加锁
-- 事务B可以插入age=25的记录

SELECT * FROM user WHERE age BETWEEN 20 AND 30 FOR UPDATE;  -- 当前读，加Next-Key Lock
-- 事务B无法插入age=25的记录（被阻塞）
```

**追问**：
- 什么是快照读和当前读？
- Next-Key Lock是如何工作的？

---

**Q4: 为什么MySQL默认隔离级别是RR而不是RC？**

**标准答案**：
主要原因是**主从复制的一致性**：

1. **MySQL 5.0之前**：binlog只支持Statement格式（记录SQL语句）
2. **RC级别问题**：在RC级别下，同一条SQL在主库和从库执行可能产生不同结果（因为可以读到其他事务的提交）
3. **RR级别优势**：保证事务内读取一致，主从复制更安全

**现在**：MySQL 5.1后支持Row格式的binlog，RC级别也可以安全使用，但为了兼容性，默认仍是RR。

**追问**：
- Statement和Row格式的binlog有什么区别？
- 什么场景下应该使用RC级别？

---

**Q5: 当前读和快照读的区别？**

**标准答案**：

| 对比 | 快照读 | 当前读 |
|------|--------|--------|
| **定义** | 读取历史版本（MVCC） | 读取最新版本（加锁） |
| **SQL** | 普通SELECT | SELECT FOR UPDATE、UPDATE、DELETE、INSERT |
| **是否加锁** | 不加锁 | 加锁（共享锁或排他锁） |
| **读取数据** | ReadView确定的版本 | 当前最新提交的版本 |
| **并发性** | 高（不阻塞） | 低（可能阻塞） |

**示例**：
```sql
-- 快照读
SELECT * FROM user WHERE id=1;  -- 读取ReadView确定的版本，不加锁

-- 当前读
SELECT * FROM user WHERE id=1 FOR UPDATE;  -- 读取最新版本，加排他锁
```

**Q12: 二叉树前序遍历如何迭代实现？**

**标准答案**：
使用栈模拟递归。先将根节点入栈，然后不断弹出栈顶节点访问，将右左子节点入栈（顺序反是为了让左节点先被处理）。

```java
public List<Integer> preorderTraversal(TreeNode root) {
    List<Integer> result = new ArrayList<>();
    Stack<TreeNode> stack = new Stack<>();
    if (root != null) stack.push(root);
    
    while (!stack.isEmpty()) {
        TreeNode node = stack.pop();
        result.add(node.val);
        if (node.right != null) stack.push(node.right);
        if (node.left != null) stack.push(node.left);
    }
    return result;
}
```

**Q13: 二叉树中序遍历如何迭代实现？**

**标准答案**：
使用栈模拟递归。一直向左遍历入栈，然后弹出访问，再向右遍历。

```java
public List<Integer> inorderTraversal(TreeNode root) {
    List<Integer> result = new ArrayList<>();
    Stack<TreeNode> stack = new Stack<>();
    TreeNode curr = root;
    
    while (curr != null || !stack.isEmpty()) {
        while (curr != null) {
            stack.push(curr);
            curr = curr.left;
        }
        curr = stack.pop();
        result.add(curr.val);
        curr = curr.right;
    }
    return result;
}
```

**Q14: 如何实现二叉树的层序遍历？**

**标准答案**：
使用队列实现BFS。关键：每次处理一层前，先记录当前队列大小（即当前层节点数）。

```java
public List<List<Integer>> levelOrder(TreeNode root) {
    List<List<Integer>> result = new ArrayList<>();
    Queue<TreeNode> queue = new LinkedList<>();
    if (root != null) queue.offer(root);
    
    while (!queue.isEmpty()) {
        int levelSize = queue.size();
        List<Integer> level = new ArrayList<>();
        for (int i = 0; i < levelSize; i++) {
            TreeNode node = queue.poll();
            level.add(node.val);
            if (node.left != null) queue.offer(node.left);
            if (node.right != null) queue.offer(node.right);
        }
        result.add(level);
    }
    return result;
}
```

**Q15: 深度优先遍历和广度优先遍历的区别？**

**标准答案**：
- DFS（深度优先）：一直往深处走，走到底再回头。用栈或递归实现，适合查看路径、拓扑排序
- BFS（广度优先）：一层一层遍历。用队列实现，适合最短路径、层序遍历

| 对比 | DFS | BFS |
|------|-----|-----|
| 实现方式 | 栈/递归 | 队列 |
| 空间复杂度 | O(h) | O(w) |
| 适用场景 | 路径、拓扑排序 | 最短路径、层序 |
