## ⚔️ CRU - Step 2: Drill (高压训练)

### 🧮 算法热身 (15min)

**题目**: LeetCode 142 - Linked List Cycle II
**链接**: https://leetcode.cn/problems/linked-list-cycle-ii/
**状态**: ✅ Pass

**题目描述**:
给定一个链表的头节点 `head`，返回链表开始入环的第一个节点。如果链表无环，则返回 `null`。不允许修改链表。

**示例**:
```
输入：head = [3,2,0,-4], pos = 1
输出：返回索引为 1 的链表节点
解释：链表中有一个环，其尾部连接到第二个节点。

链表结构：
3 -> 2 -> 0 -> -4
     ^         |
     |_________|
```

---

### 📝 算法解法详解

#### 解法一：哈希表（容易想到）

**核心思想**：遍历链表，用 HashSet 记录访问过的节点，第一个重复出现的就是环入口。

```java
public class Solution {
    public ListNode detectCycle(ListNode head) {
        Set<ListNode> visited = new HashSet<>();
        ListNode current = head;

        while (current != null) {
            if (visited.contains(current)) {
                return current;  // 这个节点之前访问过，就是环入口
            }
            visited.add(current);
            current = current.next;
        }

        return null;  // 没有环
    }
}
```

**复杂度分析**:
- 时间复杂度: O(n) - 遍历一次
- 空间复杂度: O(n) - HashSet 存储所有节点

---

#### 解法二：快慢指针（推荐，O(1) 空间）

**核心思想**：
1. 快慢指针判断是否有环，找到相遇点
2. 一个指针从头开始，一个从相遇点开始，都走一步
3. 两指针再次相遇的位置就是环入口

**数学推导**：
```
设:
  a = 头节点到环入口的距离
  b = 环入口到快慢指针相遇点的距离
  c = 环的长度
  n = 快指针在环里转的圈数

当快慢指针相遇时:
  慢指针距离 = a + b
  快指针距离 = a + b + n×c（快指针在环里多转了 n 圈）

快指针速度是慢指针的 2 倍:
  2 × (a + b) = a + b + n×c
  a + b = n×c
  a = n×c - b = (n-1)×c + (c-b)

结论: 从起点走 a 步 = 从相遇点走 (c-b) 步（都能到达环入口）
```

**图示演示**：
```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  head                                                       │
│   ↓                                                         │
│   A ──→ B ──→ C ──→ D ──→ E ──→ F                        │
│  (a)            ↑           ↓                                │
│                 (入口)     (相遇点)                             │
│                             │                                 │
│                  ┌──────────┘                                 │
│                  环 (长度 = c)                                 │
│                                                             │
│  第一阶段（找相遇点）：                                        │
│  - slow 从 A 出发，每次走 1 步                                 │
│  - fast 从 A 出发，每次走 2 步                                 │
│  - 假设在 E 点相遇                                             │
│                                                             │
│  第二阶段（找入口）：                                          │
│  - ptr1 从 A（head）开始，ptr2 从 E（相遇点）开始                   │
│  - 两个指针都每次走 1 步                                       │
│  - ptr1 和 ptr2 会在 C 点（环入口）相遇！                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**代码实现**：
```java
public class Solution {
    public ListNode detectCycle(ListNode head) {
        if (head == null || head.next == null) {
            return null;
        }

        ListNode slow = head;
        ListNode fast = head;

        // 第一阶段：找相遇点
        while (fast != null && fast.next != null) {
            slow = slow.next;
            fast = fast.next.next;

            if (slow == fast) {
                // 相遇了，进入第二阶段
                ListNode ptr1 = head;
                ListNode ptr2 = slow;

                // 第二阶段：找环入口
                while (ptr1 != ptr2) {
                    ptr1 = ptr1.next;
                    ptr2 = ptr2.next;
                }
                return ptr1;  // 环入口
            }
        }

        return null;  // 没有环
    }
}
```

**复杂度分析**:
- 时间复杂度: O(n) - 最坏遍历链表两次
- 空间复杂度: O(1) - 只用了几个指针

---

### 🔍 两种解法对比

| 解法 | 时间复杂度 | 空间复杂度 | 优点 | 缺点 |
|------|------------|------------|------|------|
| **哈希表** | O(n) | O(n) | 思路简单，容易实现 | 额外空间 |
| **快慢指针** | O(n) | O(1) | 空间最优 | 需要数学推导 |

### ⚠️ 常见错误

**1. 第二个阶段起始位置错误**
```java
// ❌ 错误代码
if (slow == fast) {
    ListNode ptr1 = head;
    ListNode ptr2 = fast;  // fast 可能为 null！

    while (ptr1 != ptr2) {
        ptr1 = ptr1.next;
        ptr2 = ptr2.next;  // NPE！
    }
}
```

**修正**：`ptr2 = slow` 或 `ptr2 = fast` 都可以，因为相遇时 `slow == fast`

**2. 边界条件未处理**
```java
// ❌ 忘记检查空链表
public ListNode detectCycle(ListNode head) {
    ListNode slow = head;
    ListNode fast = head;
    // 如果 head 为 null，fast.next 会 NPE
}
```

**3. 理解错误**
```java
// ❌ 错误理解：认为相遇点就是环入口
if (slow == fast) {
    return slow;  // 错误！相遇点不一定是环入口
}
```

#### 1.2 引用 vs 对象

```java
public void method() {
    // 基本类型 - 值直接存在栈里
    int a = 10;

    // 引用类型 - 引用在栈，对象在堆
    List<String> list = new ArrayList<>();
    // list 引用存在栈中，存的是地址（如 0x1234）
    // new ArrayList() 对象在堆中，地址是 0x1234

    // 调用方法时，通过引用找到堆中的对象
    list.add("hello");
}
```

#### 1.3 JDK 7 vs JDK 8 方法区变化

| 维度 | JDK 7 | JDK 8+ |
|------|-------|--------|
| 名称 | 永久代 | 元空间 |
| 位置 | 堆内 | 本地内存 |
| 大小限制 | 固定，容易 OOM | 默认无上限 |
| GC 影响 | 参与 GC | 基本不参与 |
| 调优参数 | -XX:PermSize/-XX:MaxPermSize | -XX:MetaspaceSize/-XX:MaxMetaspaceSize |

**为什么改？**
- 永久代大小固定，难以调优（设小了 OOM，设大了浪费）
- 动态类生成场景（Spring AOP、JSP、反射）容易 OOM
- 元空间使用本地内存，可以动态扩展，不易 OOM

#### 1.4 GC Root

**GC Root 是一组"必须活跃的对象"，作为垃圾回收的起点。**

**常见的 GC Root**：

| 类型 | 例子 |
|------|------|
| 虚拟机栈中的局部变量 | `method()` 中的 `Object localObj` |
| 方法区中的静态变量 | `private static List list` |
| 方法区中的常量 | `private static final String CONSTANT` |
| 本地方法栈中的 JNI 引用 | `native` 方法引用的对象 |

**示例代码**：
```java
public class GCRootDemo {
    // GC Root: 静态变量
    private static List<String> staticList = new ArrayList<>();

    // GC Root: 常量
    private static final String CONSTANT = "HELLO";

    public void method() {
        // GC Root: 局部变量
        Object localObj = new Object();

        // localObj 被 GC Root 引用，不会被回收
    }
}
```

### 二、GC 算法

#### 2.1 垃圾判断：可达性分析

**从 GC Root 出发，能到达的对象是活的，不能到达的是垃圾。**

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  GC Root (main 方法的局部变量)                              │
│   ↓                                                         │
│  obj1 ─────────→ obj2 ─────────→ obj3  (可达，存活)           │
│                      │                                     │
│                      ↓                                     │
│                    obj4  (可达，存活)                         │
│                                                             │
│  obj5  (不可达，是垃圾！→ 回收)                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**为什么不用引用计数法？**
- 循环引用问题：两个对象互相引用，但都无法从 GC Root 到达
- 引用计数无法识别这种情况，导致内存泄漏

---

#### 2.2 标记-清除算法 (Mark-Sweep)

**流程**：
```
第一阶段：标记
从 GC Root 出发，标记所有可达对象

第二阶段：清除
遍历堆，回收未标记的对象
```

**缺点**：
1. **内存碎片**：清除后产生大量不连续的空闲空间
2. **效率低**：标记和清除都要遍历全部对象

**图示**：
```
【清除前】A、C、E、G 活着
┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐
│ A │   │ C │   │ E │   │ G │   │   │   │   │   │
└───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘

【清除后】产生碎片
┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐
│ A │空 │ C │空 │ E │空 │ G │空 │空 │空 │空 │空 │空 │
└───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘
      ↑ 连续的空闲空间太小，大对象分配失败 → 频繁 GC
```

---

#### 2.3 复制算法 (Copying)

**流程**：
```
1. 把内存分成两块：From 和 To
2. 在 From 区分配对象
3. GC 时，把 From 区活对象复制到 To 区
4. 清空 From 区
5. From 和 To 角色互换
```

**优点**：
- 无内存碎片
- 效率高（只需遍历活对象）

**缺点**：
- 内存利用率只有 50%

**新生代优化（8:1:1）**：
```
┌─────────────────────────────────────────────────────────────┐
│  Eden (80%)    │ Survivor S0 (10%) │ Survivor S1 (10%) │
├─────────────────────────────────────────────────────────────┤
│  大对象放这里  │   存活对象复制    │   存活对象复制    │
│  大部分死      │   到这里           │   到这里           │
└─────────────────────────────────────────────────────────────┘

内存利用率：90%（Eden+S0 在用）
```

---

#### 2.4 标记-整理算法 (Mark-Compact)

**流程**：
```
1. 标记：从 GC Root 标记所有活对象
2. 整理：把活对象向一端移动，变成连续的
3. 清除：回收端部空间
```

**图示**：
```
【整理前】
┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐
│ A │空 │ C │空 │ E │空 │ G │空 │空 │空 │空 │空 │空 │
└───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘

【整理后】活对象紧凑排列
┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐
│ A │ C │ E │ G │   │   │   │   │   │   │   │   │
└───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘
     ↑ 活对象连续  ↑ 连续的空闲空间（无碎片）
```

**优点**：无碎片，内存利用率高
**缺点**：需要移动对象，效率略低

---

#### 2.5 三种算法对比

| 算法 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **标记-清除** | 简单 | 碎片、效率低 | 老年代（CMS） |
| **复制** | 无碎片、高效 | 内存浪费 50% | 新生代（8:1:1 优化后 90%） |
| **标记-整理** | 无碎片、利用率高 | 需要移动对象 | 老年代（G1） |

---

#### 2.6 CMS vs G1 收集器

| 收集器 | 新生代算法 | 老年代算法 | 缺点 |
|--------|------------|------------|------|
| **CMS** | 复制 | **标记-清除** | 有碎片，已过时 |
| **G1** | 复制 | **复制/整理** | 无碎片，JDK 9+ 默认 |

**CMS (Concurrent Mark Sweep)**：
- 老年代用**标记-清除**，有内存碎片
- 已被 G1 取代

**G1 (Garbage-First)**：
- 把堆分成多个 Region
- 回收时把活对象**复制**到别的 Region
- **本质是复制算法，无碎片**
- JDK 9+ 默认收集器

### 四、常见误区

### ❌ 误区1：栈是所有线程共享

```
正确理解：
栈是每个线程独立的！
- 线程 A 的栈和线程 B 的栈完全隔离
- 互不干扰，保证线程安全
```

### ❌ 误区2：对象一定在堆中

```
正确理解：
绝大部分对象在堆中，但有例外：
- JNI 方法中可能在本地内存分配
- 对象逃逸分析后可能在栈上分配（JIT 优化）
```

### ❌ 误区3：GC 后内存一定连续

```
正确理解：
取决于 GC 算法：
- 标记-清除：产生碎片
- 复制算法：连续
- 标记-整理：连续
```
