## 算法学习记录（框架已建，待明日继续压实）

### 1. 题目描述
- 题目：`LeetCode 198 - 打家劫舍`
- 给定一个整数数组 `nums`
- 每个元素表示一间房屋中的金额
- 相邻两间房屋不能在同一晚被偷
- 问：在不触发警报的前提下，最多能偷到多少钱

### 2. 解法一
- 解法：`动态规划`
- 最小定义：
  - 设 `dp[i]` 表示“偷到下标 `i` 这间房为止，最多能获得多少钱”
- 状态转移：
  - 不偷第 `i` 间：答案来自 `dp[i - 1]`
  - 偷第 `i` 间：答案来自 `dp[i - 2] + nums[i]`
  - 所以：`dp[i] = max(dp[i - 1], dp[i - 2] + nums[i])`

### 3. 错误思路对比
- 错误思路：看到当前房子钱多就直接偷
- 问题：这是局部最优，不等于全局最优
- 反例：`[2, 7, 9, 3, 1]`
- 如果只看眼前大的数，容易漏掉“隔一个取更优”的组合

### 4. 复杂度分析
- 时间复杂度：`O(n)`
- 空间复杂度：`O(n)`
- 可进一步优化到：`O(1)`

### 5. 常见错误
- 错误 1：`dp[i]` 的定义不清，导致转移方程乱掉
- 错误 2：边界条件没处理好，如 `n = 1` 或 `n = 2`
- 错误 3：把“不能偷相邻”误写成“只能隔两个偷”

## 技术学习记录（已完成第一轮）

### 1. 定义
- `ChannelPipeline` 可以理解成 Netty 给每个连接维护的一条“处理链”
- 链上挂着多个 `ChannelHandler`
- 不同 Handler 分别负责不同职责，比如：
  - 解码
  - 编码
  - 连接生命周期事件处理
  - 业务逻辑处理

### 2. 实际场景
- 场景：一个 RPC 服务端收到客户端发来的二进制请求
- 这个请求不会直接变成“调用 Java 方法”
- 它通常会先经历：
  - 读取字节流
  - 按协议拆包
  - 反序列化成请求对象
  - 路由到目标服务和目标方法
  - 执行业务逻辑
  - 把结果再编码回字节流写出去

### 3. Why
- 为什么需要 Pipeline：
  - 因为网络处理链天然是分步骤的
  - 如果把拆包、解码、鉴权、业务处理、编码全部堆在一个类里，职责会混乱，复用性也很差
- 为什么要分入站和出站：
  - 因为“从网络读进来”和“往网络写出去”是两条方向相反的链路
  - 解码更偏入站，编码更偏出站

### 4. 核心机制
- 最小理解先记 3 句话：
  - 入站事件：数据从网卡进来，沿着入站 Handler 往后传播
  - 出站事件：数据要写回网络，沿着出站 Handler 往前或按出站规则传播
  - Pipeline 的价值不在“有一条链”，而在“每一段链只做自己那一层的职责”

### 4.1 Pipeline 和 Handler 的关系
- `Pipeline` 是整条处理链
- `Handler` 是链上的具体处理节点
- `Pipeline` 不只是“装 Handler 的 List”
- 它更本质的作用，是承载 Netty 的事件传播模型，让不同类型的事件按规则流经不同 Handler

### 4.2 为什么 Handler 顺序不能乱
- Pipeline 本质上是一条逐步加工的数据处理链
- 前一个 Handler 的输出，往往就是后一个 Handler 的输入
- 如果顺序打乱，后面的 Handler 可能拿到自己无法识别的数据
- 例如：
  - 业务 Handler 需要的是“请求对象”
  - 但解码前拿到的还是“原始字节流”
  - 所以业务 Handler 通常不能放在解码 Handler 前面

### 4.3 Inbound / Outbound 的最小理解
- `入站`：`网络 -> Netty 内部`
- `出站`：`Netty 内部 -> 网络`
- 最常见的入站事件：
  - 连接建立
  - 收到数据
  - 读到消息
- 最常见的出站事件：
  - 写响应
  - flush
  - 关闭连接前的某些写动作

### 4.4 为什么不是每次都跑完整条 Pipeline
- 同一条 Pipeline 里，可以同时存在 `Inbound Handler`、`Outbound Handler`、`Duplex Handler`
- 但事件传播时，不是把整条链从头到尾无差别执行一遍
- 而是按“当前事件类型 + 当前传播方向”去找能够处理该事件的 Handler
- 所以：
  - 请求进入服务端时，通常走入站链
  - `Encoder` 这类出站 Handler 不会在这一步执行
  - 业务处理完调用 `writeAndFlush` 时，触发的是出站事件
  - 此时通常不会再去执行前面的解码 Handler

### 4.5 Inbound / Outbound / Duplex Handler 的职责边界
- `ChannelInboundHandler`
  - 处理入站事件
  - 典型角色：拆包器、解码器、请求读取处理器
- `ChannelOutboundHandler`
  - 处理出站事件
  - 典型角色：编码器、写回前加工逻辑
- `ChannelDuplexHandler`
  - 同时处理入站和出站事件
  - 典型场景：双向日志、链路追踪、统一拦截

### 4.6 为什么 Decoder 更像 Inbound，Encoder 更像 Outbound
- `Decoder` 处理的是从网络读进来的原始字节流，目标是把它还原成程序内部可处理的消息对象，所以通常属于 `Inbound`
- `Encoder` 处理的是程序内部待发送的响应对象，目标是把它转换成要写回网络的字节流，所以通常属于 `Outbound`

### 5. 常见误区
- 误区 1：`Pipeline` 就是一个普通 List
- 不完整。它本质上是 Netty 事件传播模型的承载结构，不只是存 Handler 的容器
- 误区 2：编解码和业务处理没有本质区别
- 不对。编解码解决的是“字节 <-> 对象”，业务处理解决的是“对象 -> 业务结果”
- 误区 3：所有 Handler 都按同一个方向执行
- 不对。入站和出站 Handler 的传播方向不同

### 6. 面试表达
- `ChannelPipeline` 可以理解为 Netty 对每个 Channel 维护的一条职责链。网络事件到来后，不是直接进入业务代码，而是沿着 Pipeline 在不同 Handler 之间传播。这样可以把拆包、编解码、鉴权、业务处理、编码回写等步骤分层解耦。对 RPC 来说，它的价值就是把“字节流 -> 请求对象 -> 方法调用 -> 响应对象 -> 字节流”这条链路拆成多个清晰阶段。

## RPC 请求处理链（已补充）

### 1. 一条 RPC 请求的最小主线
- `请求字节流进来`
- `拆包`
- `协议解析`
- `反序列化`
- `得到 RpcRequest 请求对象`
- `服务定位 + 方法调用`
- `得到 RpcResponse 响应对象`
- `序列化`
- `编码`
- `回写网络`

### 2. 为什么先拆包，再协议解析，再反序列化
- TCP 传输的是无边界字节流，不保证一次 read 就是一条完整消息
- 所以必须先拆包，先切出一条完整消息帧
- 拿到完整报文后，先解析协议头，识别：
  - 消息类型
  - 长度
  - 序列化方式
  - 请求标识
- 然后才能对消息体做反序列化，得到 `RpcRequest`

### 3. 协议解析和反序列化的区别
- `协议解析`
  - 处理的是协议头
  - 关注点是框架控制信息
  - 解决“这条消息该怎么解释”
- `反序列化`
  - 处理的是消息体
  - 关注点是业务对象
  - 解决“把二进制内容还原成请求对象”

### 4. 为什么方法调用不属于编解码阶段
- 编解码阶段关心的是：`字节流 <-> 对象`
- 方法调用阶段关心的是：`请求对象 -> 服务定位 -> 方法执行 -> 结果对象`
- 所以“找到目标方法并执行”已经不属于网络层的数据格式转换，而属于业务调用阶段

## 协议头设计（已补充）

### 1. 为什么 RPC 不能只传序列化后的参数
- 因为服务端不仅要拿到数据，还要知道：
  - 这是不是自己的协议
  - 这条消息属于哪种类型
  - 应该用哪种方式反序列化
  - 这条消息的边界在哪里
  - 这条响应对应哪次请求
- 所以 RPC 需要协议头来承载框架控制信息

### 2. 最小可用协议头字段
- `magic`
  - 快速识别当前报文是否属于本 RPC 协议
- `version`
  - 标识协议版本，支持协议升级和兼容
- `messageType`
  - 区分请求、响应、心跳、异常等不同消息类型
- `serializerType`
  - 告诉接收方消息体采用哪种序列化方式
- `requestId`
  - 关联请求与响应，支撑并发场景下的异步 RPC
- `length`
  - 标识消息长度，解决 TCP 粘包拆包问题

## LengthFieldBasedFrameDecoder（已补充）

### 1. 定义
- `LengthFieldBasedFrameDecoder` 的职责不是解析业务数据
- 它的职责是：根据协议中的 `length` 字段，从连续字节流中切出一条完整消息帧

### 2. 为什么它能解决拆包问题
- 因为它不是靠猜测边界，而是靠长度字段明确知道“一条消息应该有多长”
- 如果字节还没到齐，就继续等待
- 如果一次到了多条消息，就按长度一帧一帧切开

### 3. 它解决的两类问题
- 半包问题：
  - 当前数据不够一整帧时，先不往下解码
- 粘包问题：
  - 当前数据超过一整帧时，先切出第一帧，剩余数据继续留给下一轮切分

### 4. 为什么它是 RPC 编解码链路里的第一道门
- 因为只有先正确切出完整消息帧
- 后续的协议解析、反序列化和方法调用才有合法输入

### 5. 为什么二进制 RPC 协议更喜欢 length-based
- 二进制消息体里可能天然出现任意字节
- 如果依赖特殊分隔符，容易把正文误判成边界
- 使用 `length` 字段更稳定，也更适合变长消息体
