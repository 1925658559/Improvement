## 算法学习记录（待补）

### 1. 题目描述
- 题目：`LeetCode 207 - 课程表`
- 给定课程总数 `numCourses` 和先修关系 `prerequisites`
- 若 `[a, b]` 表示想学课程 `a`，必须先学课程 `b`
- 问：是否可以学完所有课程
- 本质：判断一张有向图中是否存在环

### 2. 解法一
- 解法：`BFS + 入度数组 + 邻接表`，本质是拓扑排序
- 建模方式：
- 每门课看成一个节点
- `[a, b]` 表示一条边 `b -> a`
- `indegree[a]++`，表示课程 `a` 多了一个前置依赖
- `graph[b].add(a)`，表示课程 `b` 学完后会影响课程 `a`
- 流程：
- 先把所有 `入度为 0` 的课程入队，表示这些课不依赖其他课程，可以先学
- 每次出队一门课，表示这门课已经完成
- 遍历它的所有后继课程，把这些课程的入度减 1，表示消除一层前置依赖
- 如果某门课的入度减到 0，就说明它的所有前置条件已经满足，可以继续入队
- 最后若“处理掉的课程数 == 总课程数”，说明图中无环，可以学完；否则说明仍有课程永远无法变成入度 0，本质上存在环

### 2.1 最小样例

- 若 `prerequisites = [[1,0],[2,0],[3,1],[3,2]]`
- 图结构为：
- `0 -> 1`
- `0 -> 2`
- `1 -> 3`
- `2 -> 3`
- 初始入度：
- `0: 0`
- `1: 1`
- `2: 1`
- `3: 2`
- 初始队列只有 `0`
- 先处理 `0` 后，`1` 和 `2` 的入度都减到 `0`
- 再处理 `1`、`2` 后，`3` 的入度减到 `0`
- 最终 4 门课都能被处理，说明无环

### 3. 解法二或错误思路对比
- 错误思路 1：把这题当成普通 BFS 层序遍历
- 问题在于这里队列的意义不是“按层遍历节点”，而是“维护当前所有可以学的课程”
- 入队条件不是“被访问到”，而是“入度降为 0”
- 错误思路 2：看到多个依赖就直接用 DFS 硬搜路径
- DFS 也能做，但如果没有建立“有向图判环”的视角，容易写成暴力回溯，思路不稳定
- 对当前阶段来说，先把 `BFS 拓扑排序` 的模式建立起来更重要

### 4. 复杂度分析
- 设课程数为 `V`，先修关系数为 `E`
- 时间复杂度：`O(V + E)`
- 每个节点最多入队出队一次
- 每条边最多被遍历一次
- 空间复杂度：`O(V + E)`
- 主要来自邻接表、入度数组和队列

### 5. 常见错误
- 错误 1：边方向写反
- `[a, b]` 不是 `a -> b`，而是 `b -> a`
- 错误 2：不知道入度表示什么
- 入度不是“有几个后继”，而是“还有几个前置依赖没满足”
- 错误 3：判断条件写错
- 不是看队列最后空不空，而是看“处理课程数是否等于总课程数”
- 错误 4：把“入度减 1”理解成纯技巧
- 本质上它表示“某个前置课程已经完成，因此消除一层依赖”

## 技术学习记录（待补）

### 1. 定义
- `boss / worker` 线程组，本质上是把 IO 事件按职责拆开：
- `boss` 主要负责 `accept`，也就是接收新连接
- `worker` 主要负责已建立连接上的 `read / write / 业务处理入口`
- 它不是“两个线程而已”，而是 `主从 Reactor` 在线程模型上的工程化落地

### 2. 实际场景
- 场景：一个 RPC 服务端，既要持续接入大量新客户端连接，又要处理已经建立连接上的请求收发
- 如果同一批线程既负责“接新连接”，又负责“处理老连接的数据读写”，那么一旦某些连接的数据处理变重，`accept` 就可能被拖慢
- 结果就是：新连接建立变慢，连接积压增加，系统看起来像“整体都卡了”

### 3. Why
- 为什么要拆：
- `接连接` 面对的是“连接建立压力”
- `管连接` 面对的是“数据收发和事件处理压力”
- 这两类压力的峰值时间、消耗资源、瓶颈位置都不同，所以不应该混在一组线程里竞争
- 拆开之后，`boss` 只做轻量且明确的连接接入，`worker` 专注处理已连接 Channel 的 IO 事件，系统吞吐更稳定

### 4. 核心机制
- 可以先把它理解成一个最小分工模型：
- 第一步：`boss` 线程阻塞/监听在服务端端口，对 `OP_ACCEPT` 事件负责
- 第二步：有新连接到来时，`boss` 把连接注册给某个 `worker`
- 第三步：之后这个连接上的 `OP_READ / OP_WRITE` 等事件，主要由对应 `worker` 负责
- 这就是为什么说 Netty 的 `bossGroup / workerGroup` 对应的是 `主从 Reactor`，而不是“随便分两个线程池”

### 4.1 关键底层动作

- 所谓“`boss` 把连接交给 `worker`”，本质不是简单地“把任务丢给另一个线程”
- 更准确的说法是：
- `boss accept 到 SocketChannel` -> `选择一个 worker` -> `把该 Channel 注册到这个 worker 的 Selector`
- 从这一刻起，这个连接后续的 `read/write` 事件，就进入该 `worker` 的事件循环管理范围
- 所以“交给 worker”这件事的关键词，不是分发，而是 `注册`

### 4.2 为什么一定要注册到 worker 的 Selector

- `worker` 不是拿到连接对象就天然能处理它
- `worker` 真正工作的入口，是自己的 `Selector + 事件循环`
- 只有把新连接注册到 `worker` 的 `Selector` 上，`worker` 才能感知这个连接什么时候可读、什么时候可写
- 因此：
- `交给 worker` = `让 worker 的事件循环正式接管这个连接后续的 IO 事件`

### 5. 常见误区
- 误区 1：`worker` 线程就是业务线程
- 不是。`worker` 首先是 IO 事件线程；如果业务逻辑很重，仍然应该把业务处理再下沉到独立业务线程池
- 误区 2：拆成 boss/worker 的核心目的是“提高并行度”
- 不完整。更本质的目标是 `职责隔离` 和 `避免不同类型压力互相拖垮`
- 误区 3：每个连接一个线程会更简单
- 简单但不经济。连接数一多，线程切换成本、内存占用、调度压力都会迅速恶化

### 6. 面试表达
- Netty 里的 `boss / worker` 可以理解为 `主从 Reactor` 在线程模型上的落地。`boss` 专门负责监听端口并接收新连接，`worker` 负责已建立连接上的读写事件处理。之所以这么拆，是因为“接连接”和“管连接”面对的是两种不同压力，如果混在同一组线程里，数据处理一重就会影响新连接接入，系统吞吐和稳定性都会下降。

### 7. 对比项：单 Reactor / 多 Reactor / 主从 Reactor

- `单 Reactor`
- 一个 Reactor 统一处理 `accept + read/write`
- 问题在于所有事件都共享同一个事件循环，只要某一段处理变慢，整体都会被拖慢
- `多 Reactor`
- 说明 Reactor 数量增加了，但不等于一定按职责拆分
- 判断关键不是“有几个 Reactor”，而是“有没有把 accept 和 read/write 分开”
- `主从 Reactor`
- 主 Reactor 负责 `accept`
- 从 Reactor 负责已建立连接上的 `read/write`
- 核心价值是 `职责隔离`
- `Netty bossGroup / workerGroup`
- 属于典型的 `主从 Reactor` 落地
- `bossGroup` 对应主 Reactor，`workerGroup` 对应从 Reactor

### 8. 风险点：为什么 worker 不等于业务线程

- `worker` 的第一身份是 `IO 事件线程`
- 它不仅处理某个 handler 里的逻辑，还要负责：
- `Selector` 轮询
- `read/write` 事件处理
- 编解码
- `ChannelPipeline` 的事件推进
- 如果把重业务长期直接放在 `worker` 上执行，会导致：
- `worker` 不能及时回到事件循环
- 它负责的其他连接 IO 也一起被拖慢
- 最终表现为读写延迟升高、请求堆积、吞吐下降、心跳超时

## 技术学习记录（已补充）

### 1. 定义
- `boss / worker` 是 Netty 对 `主从 Reactor` 的工程化实现
- `boss` 负责新连接接入
- `worker` 负责已建立连接上的 IO 事件处理

### 2. 实际场景
- 在 RPC 服务端里，一边会持续有客户端新建连接，一边已有连接会不断发生请求收发
- 这两类压力不能混在同一事件循环里，否则旧连接上的慢处理会反向拖慢新连接接入

### 3. Why
- 因为 `accept` 和 `read/write` 不是同一类压力
- 前者强调快速接入，后者强调持续处理
- 把两者拆开，系统吞吐和稳定性会更好

### 4. 核心机制
- `boss` 监听 `OP_ACCEPT`
- 新连接到来后，由 `boss` 完成 `accept`
- 然后把新连接注册到某个 `worker` 的 `Selector`
- 后续该连接上的 `read/write` 事件由该 `worker` 接管

### 5. 常见误区
- 不要把 `worker` 理解成纯业务线程
- 不要把 `主从 Reactor` 理解成“线程更多了”
- 真正关键的是 `accept` 和 `read/write` 的职责拆分

### 6. 面试表达
- Netty 服务端通常采用主从 Reactor 模型。`bossGroup` 对应主 Reactor，负责监听端口和接收新连接；`workerGroup` 对应从 Reactor，负责连接建立后的读写事件处理。这样设计是为了把连接接入压力和数据处理压力隔离开，避免某些连接上的读写或业务处理变重时，反向影响新连接接入能力。需要注意的是，`worker` 本质上是 IO 事件线程，不等于业务线程；如果把重业务直接放在 `worker` 上执行，会阻塞事件循环，拖慢它负责的其他连接。

## 学习推进补充

### 第一层结论：为什么要先拆“接连接”和“管连接”

- 先记住一句话：
- `accept` 处理的是“有没有新人进来”
- `read/write` 处理的是“已经进来的人要办什么事”
- 这两件事看起来都属于网络 IO，但在系统压力上不是一类问题
- 所以 Reactor 模型才会从 `单 Reactor` 继续演进到 `主从 Reactor`

### 最小场景类比

- 想象一个门诊前台：
- 一个人负责 `挂号接待`
- 另一批人负责 `问诊处理`
- 如果让同一个人既挂号又问诊，那么一旦某个病人沟通很久，后面排队的人连号都挂不上
- 这就是为什么系统设计里要把 `接连接` 和 `管连接` 分开
