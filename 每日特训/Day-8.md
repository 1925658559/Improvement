### 🧮 算法热身 (15min)

**题目**: LeetCode 94 - 二叉树的中序遍历
**链接**: https://leetcode.cn/problems/binary-tree-inorder-traversal/
**状态**: ✅ Pass

**题目描述**:
给你二叉树的根节点 `root`，返回它节点值的**中序遍历**。

**示例**:
```
输入：root = [4,2,6,1,3]

       4
      / \
     2   6
    / \
   1   3

输出：[1,2,3,4,6]
```

---

### 📝 算法解法详解

#### 解法一：递归实现（推荐）

**核心思想**：按照"左-根-右"的顺序递归遍历

```java
public List<Integer> inorderTraversal(TreeNode root) {
    List<Integer> result = new ArrayList<>();
    inorder(root, result);
    return result;
}

private void inorder(TreeNode node, List<Integer> result) {
    // 终止条件
    if (node == null) {
        return;
    }

    // 递归体：左 - 根 - 右
    inorder(node.left, result);    // 1. 递归遍历左子树
    result.add(node.val);          // 2. 访问根节点
    inorder(node.right, result);   // 3. 递归遍历右子树
}
```

**复杂度分析**:
- 时间复杂度: O(n) - 每个节点访问一次
- 空间复杂度: O(h) - h 为树的高度，递归栈深度

---

#### 解法二：迭代实现（使用栈）

**核心思想**：用栈模拟递归调用，先找到最左节点

```java
public List<Integer> inorderTraversalIterative(TreeNode root) {
    List<Integer> result = new ArrayList<>();
    if (root == null) {
        return result;
    }

    Stack<TreeNode> stack = new Stack<>();
    TreeNode curr = root;

    while (curr != null || !stack.isEmpty()) {
        // 1. 一直往左走，把所有左节点入栈
        while (curr != null) {
            stack.push(curr);
            curr = curr.left;
        }

        // 2. 弹出栈顶节点（最左节点）
        curr = stack.pop();
        result.add(curr.val);  // 访问根节点

        // 3. 转向右子树
        curr = curr.right;
    }

    return result;
}
```

**复杂度分析**:
- 时间复杂度: O(n)
- 空间复杂度: O(n) - 最坏情况栈存储所有节点

### 🔍 两种解法对比

| 解法 | 时间复杂度 | 空间复杂度 | 优点 | 缺点 |
|------|------------|------------|------|------|
| **递归** | O(n) | O(h) | 代码简洁，易理解 | 递归深度过大可能栈溢出 |
| **迭代** | O(n) | O(n) | 无递归开销 | 代码稍复杂 |

### ⚠️ 常见错误

**1. 递归终止条件遗漏**
```java
// ❌ 错误代码：没有检查 null
private void inorder(TreeNode node, List<Integer> result) {
    inorder(node.left, result);  // NPE！
    result.add(node.val);
    inorder(node.right, result);
}

// ✅ 正确代码
if (node == null) return;
```

**2. 混淆三种遍历顺序**
```
前序遍历：根 → 左 → 右
中序遍历：左 → 根 → 右
后序遍历：左 → 右 → 根
```

**3. 迭代实现忘记转向右子树**
```java
// ❌ 错误代码：访问完节点后没有转向右子树
curr = stack.pop();
result.add(curr.val);
// 缺少：curr = curr.right;

// ✅ 正确代码
curr = stack.pop();
result.add(curr.val);
curr = curr.right;  // 必须转向右子树
```

## 一、MySQL 索引原理

### 1.1 为什么需要索引？

**类比：查字典**

```
┌─────────────────────────────────────────────────────────────────┐
│                    查字典的两种方式                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  方式1：从第1页翻到最后一页                                      │
│    → 时间复杂度：O(n)                                           │
│    → 1000页的书需要翻1000次                                     │
│                                                                 │
│  方式2：通过目录找到页码，直接跳转                                │
│    → 时间复杂度：O(log n)                                       │
│    → 1000页的书只需要查3-4次                                    │
│                                                                 │
│  目录 = 索引                                                    │
│  关键词 → 页码 = 索引列 → 数据地址                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**数据库场景**：

| 方式 | 过程 | 时间复杂度 | 1000万数据耗时 |
|------|------|------------|---------------|
| **无索引** | 全表扫描，逐条比对 | O(n) | 约10秒 |
| **有索引** | 通过索引表直接定位 | O(log n) | 约0.001秒 |

---

### 1.2 为什么选择 B+树？

**候选数据结构对比**：

| 数据结构 | 精确查询 | 范围查询 | 排序 | 为什么不选？ |
|---------|---------|---------|------|-------------|
| **数组** | O(log n) | ✅ | ✅ | 插入/删除 O(n) |
| **哈希表** | ✅ O(1) | ❌ | ❌ | 无序，不支持范围查询 |
| **BST** | O(log n) | ✅ | ✅ | 树太高，磁盘I/O次数多 |
| **B+树** | ✅ O(log n) | ✅ | ✅ | 多叉树，树矮，I/O少 ✅ |

---

### 1.3 BST vs B+树：树高对比

**关键事实**：
- 索引存储在磁盘上
- 每访问一个节点 = 一次磁盘 I/O
- 磁盘 I/O 比内存访问慢 **10万倍**

**1000万数据的树高对比**：

| 树类型 | 每节点子节点数 | 树高 | 磁盘I/O次数 |
|--------|---------------|------|------------|
| **BST（二叉树）** | 2 | 23-24 | 23-24 次 ❌ |
| **B+树** | 1000 | 3 | 3 次 ✅ |

**B+树核心设计思想**：
```
用"空间换时间"
- 每个节点更大（存多个key）
- 树高更低（减少I/O）
```

## 二、B+树 vs B树

### 2.1 结构对比

```
┌─────────────────────────────────────────────────────────────────┐
│                        B树 vs B+树                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  B树（每个节点都存数据）：                                        │
│                                                                 │
│              [10|20|30]      ← 内部节点存key + 数据             │
│             /    |    \                                        │
│        [5|8]  [15|18]  [25|28]  ← 叶子节点也存数据              │
│                                                                 │
│  ===========================================================     │
│                                                                 │
│  B+树（只有叶子节点存数据）：                                     │
│                                                                 │
│              [10|20|30]      ← 内部节点只存key（索引）            │
│             /    |    \                                        │
│        [5|8]  [15|18]  [25|28]  ← 叶子节点存key + 数据          │
│         ↓      ↓       ↓                                         │
│       数据    数据     数据                                       │
│                                                                 │
│  + B+树叶子节点之间有指针连接（形成链表）：                        │
│                                                                 │
│    [5|8] ⇄ [15|18] ⇄ [25|28]  ← 支持范围扫描                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

---

### 2.2 B+树的核心优势

**优势1：非叶子节点只存 key**

假设一个节点大小是固定的（16KB）：
- **B树**：存 key + 数据（8字节 + 100字节）→ 只能存 150 个
- **B+树**：只存 key（8字节）→ 能存 2000 个

**结果**：同样高度的树，B+树能存更多数据

---

**优势2：叶子节点形成链表**

**范围查询场景**：`WHERE age BETWEEN 20 AND 30`

- **B树**：需要中序遍历整棵树
- **B+树**：找到 age=20 的叶子节点，沿着链表遍历到 age=30

```
B+树范围查询：
1. 找到 age=20 的叶子节点（3次I/O）
2. 沿着链表遍历（顺序I/O，非常快）
3. 直到 age=30 停止
```

---

**优势3：查询稳定**

- **B树**：数据分散在所有节点，查询路径长度不一致
- **B+树**：所有查询都走到叶子节点，性能可预测

---

### 2.3 B+树 vs B树 总结

| 特性 | B树 | B+树 |
|------|-----|------|
| 数据存储位置 | 所有节点 | 仅叶子节点 |
| 非叶子节点 | key + 数据 | 仅 key（索引） |
| 叶子节点链表 | 无 | 有（双向链表） |
| 相同高度存储量 | 较少 | 更多 ✅ |
| 范围查询 | 需要中序遍历 | 直接遍历链表 ✅ |
| 查询稳定度 | 不稳定 | 稳定（都在叶子） ✅ |

## 三、常见面试题

### Q1: 为什么 MySQL 使用 B+树而不是哈希表？

**标准答案**：
哈希表只支持精确查询（`=`），不支持范围查询（`BETWEEN`、`>`、`<`）和排序（`ORDER BY`）。B+树是有序的，支持范围查询和排序，更适合数据库场景。

### Q2: 为什么 B+树比二叉搜索树（BST）更适合做索引？

**标准答案**：
BST 是二叉树，树太高，1000万数据需要23-24次磁盘I/O。B+树是多叉树（每个节点1000个子节点），1000万数据只需3次磁盘I/O，大大减少了磁盘访问次数。

### Q3: B+树和 B树 有什么区别？

**标准答案**：
1. **数据存储**：B树所有节点都存数据，B+树只有叶子节点存数据
2. **节点容量**：B+树非叶子节点只存key，能存更多索引，树更矮
3. **范围查询**：B+树叶子节点形成链表，范围查询直接遍历链表，B树需要中序遍历

### Q4: 为什么 B+树的非叶子节点只存 key？

**标准答案**：
非叶子节点只存key，不存数据，可以在相同大小的节点中存储更多的key，从而降低树的高度，减少磁盘I/O次数。

### Q5: B+树如何支持范围查询？

**标准答案**：
B+树的叶子节点通过指针连接形成有序链表。范围查询时，先通过索引找到起始节点，然后沿着链表顺序遍历，直到结束条件，无需回溯到父节点。

## 四、常见误区

### ❌ 误区1：索引越多越好

```
正确理解：
索引会占用存储空间，并且每次插入/更新/删除数据时，都需要维护索引。
索引过多会导致写操作变慢。

建议：只对高频查询的列建立索引。
```

---

### ❌ 误区2：哈希表查询 O(1)，比 B+树 O(log n) 更快

```
正确理解：
哈希表只支持精确查询，不支持范围查询和排序。
数据库场景中，范围查询和排序非常常见，所以 B+树更适合。

MySQL 的 Memory 引擎支持哈希索引，但 InnoDB 默认使用 B+树。
```

---

### ❌ 误区3：B+树的所有节点都存数据

```
正确理解：
B+树只有叶子节点存数据，非叶子节点只存key（索引）。
这样设计可以让非叶子节点存储更多的key，降低树的高度。
```
