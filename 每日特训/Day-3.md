**题目描述**:
给你单链表的头节点 `head` ，请你反转链表，并返回反转后的链表。

**示例**:
```

```

### 📝 算法解法详解

#### 解法一：头插法（推荐）

**核心思想**：创建一个新的链表，从原链表依次摘取节点，每次插入到新链表的头部。

```
原链表: 1 -> 2 -> 3 -> 4 -> 5 -> null

过程演示：
┌─────────────────────────────────────────────────────────────┐
│  轮次 │   dummy 链表   │        cur 原链表                    │
├─────────────────────────────────────────────────────────────┤
│  初始 │ dummy -> null  │  1 -> 2 -> 3 -> 4 -> 5 -> null     │
│  第1轮│ dummy -> 1 -> null  │  2 -> 3 -> 4 -> 5 -> null     │
│  第2轮│ dummy -> 2 -> 1 -> null │  3 -> 4 -> 5 -> null      │
│  第3轮│ dummy -> 3 -> 2 -> 1 -> null │  4 -> 5 -> null      │
│  第4轮│ dummy -> 4 -> 3 -> 2 -> 1 -> null │  5 -> null      │
│  第5轮│ dummy -> 5 -> 4 -> 3 -> 2 -> 1 -> null │ null       │
└─────────────────────────────────────────────────────────────┘

返回: dummy.next = 5
```

**代码实现**:
```java
class Solution {
    public ListNode reverseList(ListNode head) {
        // 哨兵节点：简化边界处理
        ListNode dummy = new ListNode(0);
        dummy.next = null;

        ListNode cur = head;

        while (cur != null) {
            // 1. 保存下一个节点（防止断开后找不到）
            ListNode next = cur.next;

            // 2. 头插：将 cur 插入到 dummy 之后
            // cur.next 指向新链表的第一个节点
            cur.next = dummy.next;
            // dummy.next 指向 cur
            dummy.next = cur;

            // 3. 移动到原链表的下一个节点
            cur = next;
        }

        return dummy.next;
    }
}
```

**复杂度分析**:
- 时间复杂度: O(n) - 遍历一次链表
- 空间复杂度: O(1) - 只用了常数级别的额外空间

#### 解法二：三指针就地反转法

**核心思想**：直接修改原链表的指针方向，无需额外空间。

```
指针初始状态:
  null   1   ->   2   ->   3   ->   4   ->   5   -> null
   ↑     ↑
  pre   cur

第1次迭代后:
  null <- 1        2   ->   3   ->   4   ->   5   -> null
          ↑        ↑
         pre      cur

第2次迭代后:
  null <- 1   <-   2        3   ->   4   ->   5   -> null
                   ↑        ↑
                  pre      cur
```

**代码实现**:
```java
class Solution {
    public ListNode reverseList(ListNode head) {
        ListNode pre = null;   // 前驱节点
        ListNode cur = head;   // 当前节点

        while (cur != null) {
            ListNode next = cur.next;  // 保存后继节点
            cur.next = pre;            // 反转指针
            pre = cur;                 // 前驱后移
            cur = next;                // 当前后移
        }

        return pre;  // pre 指向新的头节点
    }
}
```

#### 解法三：递归法

**核心思想**：递归到链表末尾，然后逐层返回时修改指针。

```
递归调用栈:
reverseList(1)
  └─ reverseList(2)
       └─ reverseList(3)
            └─ reverseList(4)
                 └─ reverseList(5)
                      └─ reverseList(null) → return null
                 返回: 5, 5.next = 4
            返回: 5, 5.next = 3
       返回: 5, 5.next = 2
  返回: 5, 5.next = 1
```

**代码实现**:
```java
class Solution {
    public ListNode reverseList(ListNode head) {
        // Base case: 空链表或只有一个节点
        if (head == null || head.next == null) {
            return head;
        }

        // 递归反转后面的链表
        ListNode newHead = reverseList(head.next);

        // 反转当前节点和下一个节点的指针
        head.next.next = head;
        head.next = null;

        return newHead;
    }
}
```

**复杂度分析**:
- 时间复杂度: O(n)
- 空间复杂度: O(n) - 递归栈深度

---

### 🔍 三种解法对比

| 解法 | 时间复杂度 | 空间复杂度 | 优点 | 缺点 |
|------|------------|------------|------|------|
| **头插法** | O(n) | O(1) | 思路清晰，易于理解 | 需要哨兵节点 |
| **三指针反转** | O(n) | O(1) | 无额外空间，最优解 | 指针操作容易出错 |
| **递归法** | O(n) | O(n) | 代码简洁 | 递归栈溢出风险 |

### ⚠️ 常见错误

1. **忘记保存 next 指针**
```java
// ❌ 错误代码
while (cur != null) {
    cur.next = pre;  // 丢失了原链表的后续节点！
    pre = cur;
    cur = cur.next;  // NullPointerException
}
```

2. **头插法顺序错误**
```java
// ❌ 错误代码（顺序反了）
dummy.next = cur;    // 先指向 cur
cur.next = dummy.next;  // cur.next 指向自己！
```

3. **递归忘记处理 head.next**
```java
// ❌ 错误代码
public ListNode reverseList(ListNode head) {
    if (head == null) return head;
    ListNode newHead = reverseList(head.next);
    // 缺少 head.next.next = head 和 head.next = null
    return newHead;
}
```

#### Mark Word 结构（64位 JVM）

Mark Word 是对象头中最重要的部分，存储了对象的运行时数据：锁状态、hashCode、GC 年龄等。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Mark Word 64 位结构详情                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│ 【1】无锁状态 (Normal)                                                       │
│ ┌──────┬──────┬──────┬──────┬──────┐                                        │
│ │unused│ hash │unused│ age  │biased│ lock │                                   │
│ │ :25  │ :31  │ : 1  │ : 4  │ : 1  │ : 2  │                                   │
│ └──────┴──────┴──────┴──────┴──────┘                                        │
│                                                             lock: 01        │
│                                                            biased: 0        │
│                                                                             │
│ 【2】偏向锁状态 (Biased Lock)                                                 │
│ ┌───────────────────┬──────┬──────┬──────┐                                   │
│ │   thread: 54      │epoch│ age  │biased│ lock │                             │
│ │                   │ : 2  │ : 4  │ : 1  │ : 2  │                             │
│ └───────────────────┴──────┴──────┴──────┘                                   │
│                   ↑                                                          │
│            偏向线程的 ID                                              │
│                                                             lock: 01        │
│                                                            biased: 1        │
│                                                                             │
│ 【3】轻量级锁状态 (Lightweight Lock)                                           │
│ ┌───────────────────────────────┬──────┐                                    │
│ │  ptr to Lock Record: 30       │ lock │                                    │
│ │  (指向线程栈中 Lock Record)    │ : 2  │                                    │
│ └───────────────────────────────┴──────┘                                    │
│                                                             lock: 00        │
│                                                                             │
│ 【4】重量级锁状态 (Heavyweight Lock)                                          │
│ ┌───────────────────────────────┬──────┐                                    │
│ │  ptr to Monitor: 30           │ lock │                                    │
│ │  (指向堆中 ObjectMonitor)     │ : 2  │                                    │
│ └───────────────────────────────┴──────┘                                    │
│                                                             lock: 10        │
│                                                                             │
│ 【5】GC 标记                                                                 │
│ ┌───────────────────────────────┬──────┐                                    │
│ │  (标记内容，具体取决于 GC 算法) │ lock │                                    │
│ │                               │ : 2  │                                    │
│ └───────────────────────────────┴──────┘                                    │
│                                                             lock: 11        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**关键记忆**：
- 最后 2 位是锁状态标志位：`01`=无锁/偏向锁，`00`=轻量级，`10`=重量级，`11`=GC
- 无锁和偏向锁的最后 2 位都是 `01`，通过倒数第 3 位（biased 标志）区分

### 1.4 偏向锁 (Biased Lock) 详解

#### 什么是偏向锁？

**核心思想**：经过统计，大部分情况下，锁不仅没有多线程竞争，而且总是由同一个线程多次获得。

**优化方案**：当锁对象第一次被线程获取时，记录"偏向线程 ID"。之后该线程再次获取锁时，无需任何加锁操作，直接进入。

```
┌─────────────────────────────────────────────────────────────┐
│                    偏向锁工作原理                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  线程 A 第一次加锁:                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  synchronized (obj) {                               │   │
│  │      // 1. 检查 Mark Word 的 thread ID              │   │
│  │      // 2. 如果为空，CAS 设置为当前线程 ID          │   │
│  │      // 3. 标记 biased=1, lock=01                   │   │
│  │      // ... 业务逻辑                                 │   │
│  │  }                                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  线程 A 第二次加锁:                                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  synchronized (obj) {                               │   │
│  │      // 1. 检查 Mark Word 的 thread ID == 当前线程?  │   │
│  │      // 2. 是 → 直接进入！（零开销）                  │   │
│  │      // 3. 否 → 偏向撤销，升级                        │   │
│  │      // ... 业务逻辑                                 │   │
│  │  }                                                  │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 偏向锁撤销场景

| 场景 | 原因 | 结果 |
|------|------|------|
| **调用 hashCode()** | 偏向锁状态下 Mark Word 存储了 thread ID，没有空间存 hashCode | 撤销偏向 → 无锁状态，hashCode 被存储 |
| **其他线程竞争** | 偏向锁只适合单线程 | 撤销偏向 → 升级为轻量级锁 |
| **调用 wait()/notify()** | wait/notify 依赖 Monitor | 撤销偏向 → 直接升级为重量级锁 |

#### 偏向锁延迟

```
JVM 启动时会延迟偏向锁的启用（默认 4 秒）：

-XX:BiasedLockingStartupDelay=4000  # 默认 4000ms

原因：JVM 启动阶段有很多同步操作（如 System.out.println），
      如果立即启用偏向锁，会导致大量的偏向撤销，反而降低性能。
```

### 1.5 轻量级锁 (Lightweight Lock) 详解

#### 什么是轻量级锁？

**核心思想**：当有第二个线程竞争锁时，偏向锁升级为轻量级锁。轻量级锁使用 **CAS 自旋** 代替阻塞。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    轻量级锁工作原理                                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【线程栈布局】                                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  线程 A 的栈                                                          │   │
│  │  ┌─────────────────────────────────────────────────────────────┐    │   │
│  │  │  栈帧                                                         │    │   │
│  │  │  ┌───────────────────────────────────────────────────────┐  │    │   │
│  │  │  │ Lock Record (栈中分配)                                │  │    │   │
│  │  │  │  ┌─────────────────────────────────────────────────┐  │  │    │   │
│  │  │  │  │ Header: Mark Word 备份 (原始的无锁状态)          │  │  │    │   │
│  │  │  │  │ Owner: 指向锁对象                               │  │  │    │   │
│  │  │  │  └─────────────────────────────────────────────────┘  │  │    │   │
│  │  │  └───────────────────────────────────────────────────────┘  │    │   │
│  │  └─────────────────────────────────────────────────────────────┘    │   │
│  │                                                                     │   │
│  │  ┌─────────────────────────────────────────────────────────────┐    │   │
│  │  │  Lock Record 2 (如果有多个锁)                               │    │   │
│  │  └─────────────────────────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│           ▲                                                                  │
│           │ CAS 替换                                                         │
│           │                                                                  │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  锁对象 obj                                                          │   │
│  │  Mark Word: ptr to Lock Record | lock:00                           │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【加锁流程】                                                                │
│  1. 在线程栈中分配 Lock Record，备份对象原始 Mark Word                       │
│  2. CAS 尝试将 obj 的 Mark Word 替换为指向 Lock Record 的指针                │
│     - 成功 → 获得轻量级锁                                                    │
│     - 失败 → 说明有竞争，自旋重试                                            │
│                                                                             │
│  【解锁流程】                                                                │
│  1. CAS 将 Mark Word 恢复为原始值（从 Lock Record 的 Header 复制）            │
│  2. 成功 → 完全释放锁                                                        │
│  3. 失败 → 说明有其他线程竞争，唤醒阻塞线程                                  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 为什么 Lock Record 在栈中？

| 存储位置 | 优点 | 缺点 |
|----------|------|------|
| **栈** | 线程私有，无需同步；方法结束自动释放；CPU 缓存友好 | 每个线程都要分配 |
| 堆 | 共享 | 需要同步；GC 压力 |
| 全局 | 统一管理 | 互斥开销大 |

### 1.6 自旋锁与自适应自旋

#### 什么是自旋？

```
┌─────────────────────────────────────────────────────────────┐
│                    自旋 vs 阻塞                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【自旋】（轻量级锁）                                         │

│  缺点: 浪费 CPU                                              │

```

- 如果上次成功，增加自旋次数
- 如果上次失败，减少自旋次数

### 1.7 重量级锁 (Heavyweight Lock) 详解

#### ObjectMonitor 结构

```cpp
// OpenJDK 源码位置: src/hotspot/share/runtime/objectMonitor.hpp

class ObjectMonitor {
    private:
    // Mark Word 备份
    volatile intptr_t _header;

    // 指向关联的 Java 对象
    void* volatile _object;

    // 🔥 持有锁的线程（关键字段）
    void volatile _owner;

    // 🔥 竞争队列（尝试获取锁但失败的线程）
    ObjectWaiter* volatile _EntryList;

    // 🔥 等待队列（调用了 wait() 的线程）
    ObjectWaiter* volatile _WaitSet;

    // 🔥 重入次数
    volatile intptr_t _recursions;

    // 竞争队列的备用链表
    ObjectWaiter* volatile _cxq;
};
```

### 1.9 常见面试题

#### Q1: synchronized 锁升级过程是怎样的？

**标准答案**：
```
无锁 (01,0) → 偏向锁 (01,1) → 轻量级锁 (00) → 重量级锁 (10)

详细解释：
1. 无锁状态：对象刚创建，未被加锁
2. 偏向锁：第一个线程加锁时，Mark Word 存储线程 ID，无真正加锁操作
3. 轻量级锁：第二个线程竞争时，撤销偏向，升级为轻量级锁（CAS 自旋）
4. 重量级锁：自旋超过阈值或竞争激烈时，升级为重量级锁（阻塞）
```

#### Q2: 偏向锁什么时候会撤销？

**标准答案**：
1. 调用 `hashCode()` 方法（Mark Word 需要存储 hashCode）
2. 其他线程尝试获取锁（偏向锁只适合单线程）
3. 调用 `wait()/notify()` 方法（需要 Monitor）

#### Q3: 为什么 WaitSet 和 EntryList 分开？

**标准答案**：
- **EntryList**：等待获取锁的线程（BLOCKED 状态）
- **WaitSet**：调用了 `wait()` 等待被唤醒的线程（WAITING 状态）

分开的原因：
1. 语义不同：一个是竞争锁，一个是等待条件
2. 性能优化：`notify()` 只需唤醒 WaitSet 中的一个线程，而不是所有竞争线程

#### Q4: 轻量级锁为什么比重量级锁快？

**标准答案**：
- 轻量级锁使用 CAS 自旋，在用户态完成（~10 cycles）
- 重量级锁涉及用户态↔内核态切换（~1000 cycles）
- 性能差距约 100 倍

### 1.10 常见误区

#### ❌ 误区1：偏向锁一定比轻量级锁快

```
正确理解：
- 单线程场景：偏向锁最快（几乎零开销）
- 多线程交替执行：轻量级锁更快（避免偏向撤销成本）
- 多线程并发竞争：重量级锁（虽然慢，但保证正确性）
```

#### ❌ 误区2：synchronized 可重入需要手动处理

```
正确理解：
synchronized 是可重入锁！_recursions 字段记录重入次数。

public synchronized void methodA() {
    methodB();  // 可重入，不会死锁
}

public synchronized void methodB() {
    // ...
}
```

#### ❌ 误区3：关闭偏向锁一定提升性能

```
正确理解：
- 单线程场景：偏向锁能显著提升性能
- 高并发场景：关闭偏向锁可能更优（避免偏向撤销成本）

需要根据实际场景测试！
```

### 2.3 内存屏障 (Memory Barrier)

#### 什么是内存屏障？

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    内存屏障的作用                                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【定义】                                                                   │
│  内存屏障是一种 CPU 指令，用于:                                              │

#### 四种内存屏障

```

├─────────────────────────────────────────────────────────────────────────────┤

│  │                                                                      │  │

│  │                                                                      │  │
│  │  作用: 确保 Load1 读取的数据先于 Store2 及之后的写操作刷新到内存      │  │
│  │  禁止: Store2 及之后的写操作重排序到 Load1 之前                       │  │
│  └─────────────────────────────────────────────────────────────────────┘  │
│                                                                             │

### 2.4 Happens-Before 原则

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    JSR-133 Happens-Before 规则                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【定义】                                                                   │
│  如果操作 A happens-before 操作 B，那么 A 的结果对 B 可见，且 A 的执行顺序  │
│  先于 B。                                                                   │
│                                                                             │
│  【8 条规则】                                                                │
│                                                                             │
│  1. 程序顺序规则 (Program Order Rule)                                       │
│     ┌───────────────────────────────────────────────────────────────────┐  │
│     │  单线程内，代码顺序执行，书写在前面的操作 happens-before 书写在后面 │  │
│     │  的操作                                                            │  │
│     │                                                                   │  │
│     │  int x = 1;  // A happens-before B                               │  │
│     │  int y = 2;  // B                                                │  │
│     └───────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  2. 监视器锁规则 (Monitor Lock Rule)                                       │
│     ┌───────────────────────────────────────────────────────────────────┐  │
│     │  unlock 操作 happens-before 同一把锁的 lock 操作                   │  │
│     │                                                                   │  │
│     │  线程A: synchronized(obj) { x = 1; }  // unlock                   │  │
│     │  线程B: synchronized(obj) { // 读取 x  }  // lock                 │  │
│     │  // 线程B 一定能看到 x=1                                          │  │
│     └───────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  3. Volatile 变量规则 (Volatile Variable Rule)                              │
│     ┌───────────────────────────────────────────────────────────────────┐  │
│     │  volatile 写 happens-before volatile 读                          │  │
│     │                                                                   │  │
│     │  线程A: volatileVar = 1;  // volatile 写                         │  │
│     │  线程B: int x = volatileVar;  // volatile 读                     │  │
│     │  // 线程B 一定能看到 volatileVar=1                               │  │
│     └───────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  4. 线程启动规则 (Thread Start Rule)                                       │
│     ┌───────────────────────────────────────────────────────────────────┐  │
│     │  Thread.start() happens-before 线程中的每个操作                   │  │
│     │                                                                   │  │
│     │  x = 1;                                                          │  │
│     │  thread.start();  // 线程中一定能看到 x=1                        │  │
│     └───────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  5. 线程终止规则 (Thread Termination Rule)                                 │
│     ┌───────────────────────────────────────────────────────────────────┐  │
│     │  线程中的每个操作 happens-before Thread.join()                    │  │
│     │                                                                   │  │
│     │  // 线程A执行                                                     │  │
│     │  x = 1;                                                          │  │
│     │  thread.join();  // 一定能看到 x=1                               │  │
│     └───────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  6. 线程中断规则 (Thread Interruption Rule)                                 │
│     ┌───────────────────────────────────────────────────────────────────┐  │
│     │  thread.interrupt() happens-before 检测到中断事件                 │  │
│     │                                                                   │  │
│     │  thread.interrupt();                                             │  │
│     │  if (Thread.interrupted()) { ... }  // 一定能检测到中断           │  │
│     └───────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  7. 对象终结规则 (Finalizer Rule)                                          │
│     ┌───────────────────────────────────────────────────────────────────┐  │
│     │  构造函数执行结束 happens-before finalize() 开始                   │  │
│     └───────────────────────────────────────────────────────────────────┘  │
│                                                                             │
│  8. 传递性 (Transitivity)                                                  │
│     ┌───────────────────────────────────────────────────────────────────┐  │
│     │  A happens-before B，B happens-before C → A happens-before C      │  │
│     │                                                                   │  │
│     │  x = 1;        // A                                               │  │
│     │  volatile v = 2; // B (A happens-before B)                       │  │
│     │  y = 3;        // C (B happens-before C)                         │  │
│     │  // 结论: x=1 对 y=3 可见                                         │  │
│     └───────────────────────────────────────────────────────────────────┘  │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.7 常见面试题

#### Q1: Volatile 的作用是什么？

**标准答案**：
1. **可见性**：volatile 变量的写操作立即刷新到主内存，读操作从主内存读取
2. **有序性**：通过内存屏障禁止指令重排序
3. **原子性有限**：仅保证单个读/写操作的原子性，不保证复合操作（如 count++）

#### Q2: Volatile 能保证原子性吗？

**标准答案**：
不能完全保证。Volatile 仅保证：
- long/double 的赋值操作（JDK 1.5+）
- 单个变量的读写操作

对于复合操作（如 count++），需要使用 AtomicInteger 或 synchronized。

#### Q3: 为什么 DCL 单例必须加 volatile？

**标准答案**：
`instance = new Singleton()` 分为 3 步：
1. 分配内存空间
2. 初始化对象
3. 将 instance 指向内存地址

如果没有 volatile，可能发生重排序 1→3→2：
- 线程A 执行完步骤3（instance != null，但对象未初始化）
- 线程B 判断 instance != null，返回未初始化的对象

Volatile 禁止 3 和 2 重排序，确保对象完全初始化后才可见。

#### Q4: Volatile 和 Synchronized 的区别？

**标准答案**：
| 维度 | Volatile | Synchronized |
|------|----------|--------------|
| 可见性 | ✅ | ✅ |
| 有序性 | ✅ | ✅ |
| 原子性 | ❌ 仅单读写 | ✅ 完整 |
| 阻塞线程 | ❌ | ✅ |
| 性能 | 高 | 较低 |

### 2.8 常见误区

#### ❌ 误区1: Volatile 变量是线程安全的

```
正确理解：
Volatile 只保证可见性和有序性，不保证复合操作的原子性。

volatile int count = 0;
count++;  // ❌ 非线程安全！

正确做法：
- 使用 AtomicInteger（CAS）
- 使用 synchronized（锁）
```

#### ❌ 误区2: 所有变量都应该加 volatile

```
正确理解：
Volatile 有性能开销（每次都从主内存读写，禁用 CPU 缓存）。

只在以下场景使用：
1. 状态标记量（boolean flag）
2. 单例 DCL
3. 单写多读的场景
```

#### ❌ 误区3: Volatile 比 synchronized 快

```
正确理解：
- 单次读写：Volatile 更快（无锁）
- 复合操作：Volatile 不适用，必须用 synchronized 或 AtomicInteger

需要根据实际场景选择：
- 简单状态标记 → Volatile
- 复合操作（count++）→ AtomicInteger / synchronized
```
