## 技术学习记录（已完成）

### 1. 定义
- `注册中心` 维护的是 `serviceName -> 可用服务地址列表`
- `watch / 订阅` 维护的是客户端本地缓存的新鲜度
- `负载均衡` 负责从最新地址列表中选择本次请求的目标节点

### 2. 实际场景
- 客户端调用 `UserService.queryById()` 时，并不知道服务部署在哪台机器
- 如果没有注册中心，服务地址只能硬编码在客户端配置中；一旦扩容、缩容、迁移或故障，配置就会迅速失效
- 如果客户端每次都临时访问注册中心查地址，调用链会从 `consumer -> provider` 退化成 `consumer -> registry -> consumer -> provider`

### 3. Why
- 为什么不能每次都查注册中心：
  - 每次调用都会多一次网络开销，增加延迟
  - 注册中心会变成高频热点，承担不该承担的数据面压力
  - 调用会强依赖注册中心在线，降低整体可用性
- 为什么需要本地缓存 + watch：
  - 本地缓存负责承接高频调用
  - watch 负责在节点上下线时通知客户端刷新缓存
  - 这样即使注册中心短暂抖动，客户端也仍然可以基于缓存继续发请求

### 4. 核心机制
- 客户端最小链路：
  - 根据 `serviceName` 第一次拉取可用节点列表
  - 缓存在本地
  - 注册 `watch / 订阅`
  - 节点上下线时接收变更通知并刷新本地缓存
  - 负载均衡从最新的本地节点列表中选出本次目标机器
- `watch` 更新的对象：
  - 不是“本次请求要打到哪台机器”
  - 而是“当前哪些机器可用”
  - 本质是更新 `serviceName -> 地址列表`
- `轮询` 的目标：
  - 尽量把请求均匀分散到不同机器上
  - 适合节点能力和请求成本比较接近的场景
- `一致性 Hash` 的目标：
  - 让同一业务 key 的请求尽量稳定落到同一节点
  - 常见 key：`userId`、`orderId`、`sessionId`
  - 更适合需要提升本地缓存命中率、减少状态迁移或保持会话黏性的 RPC 场景
- 为什么不是普通 Hash：
  - 节点增减时，普通 Hash 会导致大量 key 重新映射
  - 一致性 Hash 只让一部分 key 迁移，更适合实例动态上下线的分布式场景
- 一致性 Hash 中节点为什么会“负责一段区间”：
  - key 做完 Hash 后通常不会正好落在某个节点点位上
  - 规则是顺时针找到第一个节点
  - 所以一个节点实际负责的是“前一个节点到自己之间”的那段区间
- 为什么真实节点过少时容易负载倾斜：
  - 节点在环上的落点可能不均匀
  - 落点不均会导致不同节点负责的区间大小不同
  - 区间越大，落入的 key 通常越多，该节点更容易更忙
- `虚拟节点` 的作用：
  - 把一台真实机器映射成环上的多个分散点
  - 本质上是把原本可能很大的连续区间切碎成很多小区间
  - 这样每台机器拿到的是分散的小段总和，从而缓解负载倾斜
- 虚拟节点为什么不会破坏稳定路由：
  - key 先命中的是某个虚拟节点
  - 但虚拟节点最终仍然隶属于某台固定的真实机器
  - 它改变的是分布粒度，不改变最终归属规则
- 为什么一致性 Hash 的重映射是局部的：
  - 节点上下线只会改变该节点及其相邻区间的归属关系
  - 其他区间仍然按原来的顺时针规则命中原节点
  - 所以只会影响部分 key，而不是全量重算
- Zookeeper `watch` 的一次性语义：
  - `watch` 是一次性通知机制，不是永久订阅
  - 事件触发一次后就失效
  - 如果客户端还想继续监听，就需要重新注册
- 用 Zookeeper 做服务发现时客户端为什么更复杂：
  - 收到通知后要重新拉取最新地址列表
  - 需要重新注册 `watch`
  - 还要处理连接断开、重连后的监听恢复
  - 本质上是客户端自己维护“持续监听 + 列表刷新”这条链路
- Zookeeper 与 Nacos 的抽象层差异：
  - Zookeeper 给的是底层节点、临时节点和节点变化能力
  - Nacos 直接给的是服务、实例、健康状态和实例列表变更这些服务发现语义
  - 所以 Nacos 更像现成的服务发现中心，Zookeeper 更像注册中心基础设施

### 5. 常见误区
- 误区 1：注册中心应该参与每次业务请求的实时路由
- 不对。注册中心更适合做控制面，负责服务发现和变更通知
- 误区 2：`watch` 决定本次请求发给哪台机器
- 不对。`watch` 维护的是可用节点集合，真正的节点选择由负载均衡完成
- 误区 3：一致性 Hash 主要是为了“处理更重的请求”
- 不对。它更核心的价值是“同 key 稳定路由 + 降低节点变更时的重映射成本”
- 误区 4：一致性 Hash 天然就能把流量分得很均匀
- 不对。真实节点较少时仍可能因为落点不均导致负载倾斜，所以工程上通常会引入虚拟节点
- 误区 5：Zookeeper 的 `watch` 注册一次后就会一直推送
- 不对。它是一次性触发机制，事件发生后需要重新注册
- 误区 6：Zookeeper 和 Nacos 只是 API 形式不同
- 不对。它们的抽象层级不同，前者更偏底层协调能力，后者更偏服务发现语义

### 6. 面试表达
- RPC 客户端通常不会每次请求都实时访问注册中心，而是先根据 `serviceName` 拉取并缓存可用节点列表，再通过 `watch / 订阅` 机制感知服务实例上下线，及时刷新本地缓存。这样可以把注册中心从高频调用路径中拿出去，降低额外网络开销，也避免注册中心成为单点热点。
- 在负载均衡层，`轮询` 的目标是把请求尽量平均分散到各节点；`一致性 Hash` 的目标则是让同一业务 key 的请求尽量稳定落到同一台机器，从而提升本地缓存命中率、减少状态迁移。它特别适合按 `userId`、`orderId`、`sessionId` 做路由的 RPC 场景。
- 一致性 Hash 中，节点负责的是前一个节点到自己之间的区间。真实节点较少且落点不均时，不同节点负责的区间大小会不同，导致负载倾斜；虚拟节点通过把真实节点映射成环上的多个分散点，把大区间切碎成小区间，从而缓解负载不均，同时不破坏同 key 稳定路由和局部重映射的特性。
- 在服务发现实现上，Zookeeper 提供的是底层节点、临时节点和一次性 `watch` 能力，适合做注册中心基础设施；但客户端需要自己把节点变化翻译成服务实例列表变化，并维护缓存刷新、`watch` 续订和异常恢复。Nacos 直接面向服务注册与发现，提供服务、实例、健康状态和实例列表变更等更高层抽象，因此更像开箱即用的服务发现中心。
