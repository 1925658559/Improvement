## ⚔️ CRU - Step 2: Drill (高压训练)

### 🧮 算法热身 (15min)

**题目**: LeetCode 1 - Two Sum
**链接**: https://leetcode.cn/problems/two-sum/
**状态**: ✅ Pass

**题目描述**:
给定一个整数数组 `nums` 和一个整数目标值 `target`，请你在该数组中找出和为目标值 `target` 的那两个整数，并返回它们的数组下标。

**示例**:
```
输入: nums = [2,7,11,15], target = 9
输出: [0,1]
解释: 因为 nums[0] + nums[1] == 9 ，返回 [0, 1] 。
```

---

### 📝 算法解法详解

#### 解法一：暴力枚举

**核心思想**：双重循环，枚举所有两两组合

```java
class Solution {
    public int[] twoSum(int[] nums, int target) {
        for (int i = 0; i < nums.length; i++) {
            for (int j = i + 1; j < nums.length; j++) {
                if (nums[i] + nums[j] == target) {
                    return new int[]{i, j};
                }
            }
        }
        return new int[0];
    }
}
```

**复杂度分析**:
- 时间复杂度: O(n²)
- 空间复杂度: O(1)

---

#### 解法二：HashMap 优化（推荐）

**核心思想**：边遍历边存，用 HashMap 存储已遍历的值及其下标

```
遍历过程演示 (nums = [2,7,11,15], target = 9):
┌─────────────────────────────────────────────────────────────┐
│  轮次 │  nums[i]  │  complement │  HashMap 状态 │   结果     │
├─────────────────────────────────────────────────────────────┤
│  i=0  │     2     │     7       │   {}          │  7不存在   │
│      │           │             │   存入2→0     │            │
│  i=1  │     7     │     2       │   {2:0}       │  ✅找到2!  │
│      │           │             │               │  返回[0,1]  │
└─────────────────────────────────────────────────────────────┘
```

**代码实现**:
```java
class Solution {
    public int[] twoSum(int[] nums, int target) {
        Map<Integer, Integer> map = new HashMap<>();

        for (int i = 0; i < nums.length; i++) {
            int complement = target - nums[i];

            // 先检查 complement 是否已在 map 中
            if (map.containsKey(complement)) {
                return new int[]{map.get(complement), i};
            }

            // 再将当前值存入 map（避免自己匹配自己）
            map.put(nums[i], i);
        }

        return new int[0];
    }
}
```

**复杂度分析**:
- 时间复杂度: O(n) - 只需遍历一次
- 空间复杂度: O(n) - HashMap 最多存储 n 个元素

---

### 🔍 两种解法对比

| 解法 | 时间复杂度 | 空间复杂度 | 优点 | 缺点 |
|------|------------|------------|------|------|
| **暴力枚举** | O(n²) | O(1) | 无额外空间 | 时间复杂度高 |
| **HashMap** | O(n) | O(n) | 时间最优 | 需要额外空间 |

### ⚠️ 常见错误

1. **自己匹配自己**
```java
// ❌ 错误代码
for (int i = 0; i < nums.length; i++) {
    map.put(nums[i], i);  // 先存入
    if (map.containsKey(target - nums[i])) {  // 再检查
        return new int[]{i, map.get(target - nums[i])};
        // 当 target = nums[i] * 2 时，会返回 [i, i]
    }
}
```

2. **边界条件处理**
```java
// ❌ 忘记处理无解情况
public int[] twoSum(int[] nums, int target) {
    // ...
    // 没有找到时，应该返回 new int[0] 或 throw 异常
}
```

### 1.3 为什么查找平均是 O(1)？

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    HashMap 查找流程                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【查找流程】                                                                │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  步骤1: key → hash() 函数 → 计算出 hash 值                           │   │
│  │         ↓                                                            │   │
│  │       (扰动函数: h ^ (h >>> 16)，减少冲突)                            │   │
│  │         ↓                                                            │   │
│  │  步骤2: hash & (n-1) → 计算出数组下标                                 │   │
│  │         ↓                                                            │   │
│  │       (位运算，等价于 hash % n，但效率更高)                            │   │
│  │         ↓                                                            │   │
│  │  步骤3: 定位到 table[index]                                          │   │
│  │         ↓                                                            │   │
│  │  步骤4: 如果该位置只有一个元素 → 直接返回 ✅                           │   │
│  │         如果有多个元素（链表/红黑树） → 遍历查找                       │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【为什么平均 O(1)？】                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  核心原因: hash 函数让你不需要遍历整个数组，直接跳到对应位置            │   │
│  │                                                                     │   │
│  │  在以下条件下，平均查找时间接近 O(1):                               │   │
│  │  - hash 函数设计良好（元素均匀分布）                                 │   │
│  │  - 负载因子合理（0.75，冲突少）                                       │   │
│  │  - 数组长度足够大                                                    │   │
│  │                                                                     │   │
│  │  在这种情况下，每个数组位置的链表长度很短（通常 < 3），遍历耗时    │   │
│  │  可忽略                                                             │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【最坏情况】                                                               │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  如果所有 key 的 hash 值都冲突 → 全部挂在同一个链表                │   │
│  │  → 退化成链表查找 → O(n)                                            │   │
│  │                                                                     │   │
│  │  这就是为什么 JDK 1.8 引入红黑树！                                   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.4 Hash 冲突处理：拉链法

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Hash 冲突与拉链法                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【什么是 Hash 冲突？】                                                     │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  两个不同的 key 计算出相同的数组下标                                   │   │
│  │                                                                     │   │
│  │  例: hash("name") = 5, hash("age") = 5                              │   │
│  │      两个 key 都要存放在 table[5]                                   │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【拉链法处理】                                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  冲突的元素用链表挂在同一个数组位置                                   │   │
│  │                                                                     │   │
│  │  table[5] → Node("name", "张三") → Node("age", 18) → Node(...)     │   │
│  │                                                                     │   │
│  │  JDK 1.7: 新元素插入到链表头部（头插法）                              │   │
│  │  JDK 1.8: 新元素插入到链表尾部（尾插法）                              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【为什么不用开放定址法？】                                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  开放定址法: 冲突了就找下一个空位                                    │   │
│  │    table[5] 有冲突 → 尝试 table[6] → table[7] ...                   │   │
│  │                                                                     │   │
│  │  缺点:                                                              │   │
│  │  - 容易聚集（连续占用），性能下降                                    │   │
│  │  - 删除操作复杂（不能直接删除，需要标记）                              │   │
│  │  - 不适合存储大量数据                                                │   │
│  │                                                                     │   │
│  │  HashMap 要存大量数据，所以用拉链法                                 │   │
│  │  ThreadLocalMap 用开放定址法（数据量小）                              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.5 JDK 1.7 头插法死循环问题

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    JDK 1.7 头插法死循环原理                                    │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【触发条件】                                                                │
│  多线程同时扩容（resize）                                                     │
│                                                                             │
│  【死循环原理】                                                              │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  假设扩容前数组长度 8 → 16，index=2 的链表:                           │   │
│  │                                                                     │   │
│  │  扩容前:  A → B → C                                                  │   │
│  │                                                                     │   │
│  │  ┌─────────────────────────────────────────────────────────────┐    │   │
│  │  │  线程1: 准备迁移 A → B，暂停（在处理 A 时）                     │    │   │
│  │  │         保存 A.next = B                                       │    │   │
│  │  ├─────────────────────────────────────────────────────────────┤    │   │
│  │  │  线程2: 完成整个迁移                                            │    │   │
│  │  │         头插法迁移后: C → B → A（倒序）                          │    │   │
│  │  │         A.next 已被线程2改为 null（或者指向新位置）              │    │   │
│  │  ├─────────────────────────────────────────────────────────────┤    │   │
│  │  │  线程1: 继续执行，持有旧引用（A.next = B）                      │    │   │
│  │  │         把 C 插入头部                                           │    │   │
│  │  │         此时 B.next 可能还指向 A（线程2修改前的状态）            │    │   │
│  │  │         形成环: A → B → A                                        │    │   │
│  │  └─────────────────────────────────────────────────────────────┘    │   │
│  │                                                                     │   │
│  │  结果:                                                              │   │
│  │    C → B → A                                                        │   │
│  │         ↑___↓                                                        │   │
│  │      (死循环)                                                        │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【JDK 1.8 尾插法为什么解决？】                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  尾插法保持元素相对顺序不变:                                         │   │
│  │  扩容前: A → B → C                                                  │   │
│  │  扩容后: A → B → C（顺序不变）                                       │   │
│  │                                                                     │   │
│  │  不会倒序，就不会形成环                                              │   │
│  │                                                                     │   │
│  │  注意: JDK 1.8 仍然不是线程安全的！只是解决了扩容死循环问题        │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.6 红黑树转换机制

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    红黑树转换条件与原理                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【转换条件（两个同时满足）】                                               │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  if (链表长度 >= 8 && 数组长度 >= 64) {                             │   │
│  │      转红黑树;                                                      │   │
│  │  } else if (链表长度 >= 8 && 数组长度 < 64) {                      │   │
│  │      优先扩容数组;                                                   │   │
│  │  }                                                                 │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【为什么链表长度 ≥ 8 才转？】                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  基于**泊松分布**数学原理                                           │   │
│  │                                                                     │   │
│  │  负载因子 0.75 时，链表长度为 k 的概率 P:                            │   │
│  │  ┌───────────────────────────────────────────────────────────┐     │   │
│  │  │  k=0: 0.606530    (60.65%)                                     │     │   │
│  │  │  k=1: 0.303265    (30.33%)                                     │     │   │
│  │  │  k=2: 0.075816    (7.58%)                                      │     │   │
│  │  │  k=3: 0.012636    (1.26%)                                      │     │   │
│  │  │  k=4: 0.001579    (0.16%)                                      │     │   │
│  │  │  k=5: 0.000157    (0.016%)                                     │     │   │
│  │  │  k=6: 0.000013    (0.0013%)                                    │     │   │
│  │  │  k=7: 0.000001    (0.0001%)                                    │     │   │
│  │  │  k=8: 0.00000006  (0.000006%) ← 千万分之六！                     │     │   │
│  │  └───────────────────────────────────────────────────────────┘     │   │
│  │                                                                     │   │
│  │  结论: 如果链表长度达到 8，说明 hash 函数质量差（大量冲突）         │   │
│  │        必须用红黑树救场                                              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【为什么还需要数组长度 ≥ 64？】                                           │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  避免无意义的转换                                                    │   │
│  │                                                                     │   │
│  │  如果数组很小（比如 16），hash 很差，所有元素撞到同一个位置:        │   │
│  │  - 此时应该优先**扩容数组**，而不是转红黑树                         │   │
│  │  - 红黑树节点空间是链表的 2 倍，频繁转换浪费性能                   │   │
│  │                                                                     │   │
│  │  示例:                                                             │   │
│  │  数组长度 16，所有元素都在 table[0]                                  │   │
│  │  → 链表长度 100                                                     │   │
│  │  → 应该扩容到 32，让元素重新分布，而不是转红黑树                     │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【红黑树退回链表】                                                          │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  当红黑树节点数 ≤ 6 时，退回链表                                    │   │
│  │                                                                     │   │
│  │  为什么是 6 而不是 8？                                              │   │
│  │  - 避免频繁转换（链表↔红黑树）                                       │   │
│  │  - 8 → 6: 有 2 个节点的缓冲区                                       │   │
│  │  - 如果都是 8，可能在 7、8 之间来回切换                             │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.7 hash() 扰动函数

```java
// JDK 1.8 的 hash 函数
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    扰动函数原理                                               │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【为什么 h ^ (h >>> 16)？】                                                │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  高位参与运算: hashCode() 的高 16 位和低 16 位做异或                │   │
│  │  减少冲突，让 hash 值更均匀地分布在数组中                            │   │
│  │                                                                     │   │
│  │  示例: 假设数组长度 16（二进制 10000）                               │   │
│  │  ┌─────────────────────────────────────────────────────────────┐    │   │
│  │  │  如果只用低位: 只有低 4 位参与运算，容易冲突                 │    │   │
│  │  │                                                             │    │   │
│  │  │  hash1 = 0x0000AAAA  (低4位都是1010)                         │    │   │
│  │  │  hash2 = 0xFFFFAAAA  (低4位都是1010)                         │    │   │
│  │  │  → 都映射到 index = 10，冲突！                                 │    │   │
│  │  │                                                             │    │   │
│  │  │  高低位异或后:                                                 │    │   │
│  │  │  hash1 ^ (hash1 >>> 16) = 0x0000AAAA                         │    │   │
│  │  │  hash2 ^ (hash2 >>> 16) = 0xFFFF5555                         │    │   │
│  │  │  → 分别映射到不同位置，冲突减少！                               │    │   │
│  │  └─────────────────────────────────────────────────────────────┘    │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【扰动前后对比】                                                            │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  假设数组长度 16，测试 10000 个随机 key:                             │   │
│  │                                                                     │   │
│  │  无扰动: hash % 16          → 冲突率 ~20%                           │   │
│  │  有扰动: hash ^ (hash>>>16) → 冲突率 ~5%                            │   │
│  │                                                                     │   │
│  │  扰动函数显著降低冲突率！                                           │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.8 计算数组下标：为什么用 `& (n-1)`？

```java
int index = hash & (n - 1);
```

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    位运算 vs 取模运算                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【为什么不用 hash % n？】                                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  % 是取模运算，涉及除法，效率低                                       │   │
│  │  & 是位运算，CPU 直接执行，效率高                                    │   │
│  │                                                                     │   │
│  │  性能对比:                                                          │   │
│  │  取模: ~10-20 CPU cycles                                            │   │
│  │  位运算: ~1 CPU cycle                                               │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【为什么是 (n-1)？】                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  HashMap 数组长度永远是 2 的幂次方（16, 32, 64...）                  │   │
│  │                                                                     │   │
│  │  2 的幂次方减 1 的二进制全是 1:                                     │   │
│  │  ┌─────────────────────────────────────────────────────────────┐     │   │
│  │  │  16 - 1 = 15  = 00001111                                        │     │   │
│  │  │  32 - 1 = 31  = 00011111                                        │     │   │
│  │  │  64 - 1 = 63  = 00111111                                        │     │   │
│  │  └─────────────────────────────────────────────────────────────┘     │   │
│  │                                                                     │   │
│  │  & 操作相当于取 hash 的低 n 位:                                     │   │
│  │  hash & (n-1) = hash 的低 log2(n) 位                               │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【示例】                                                                   │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  假设 n = 16, n-1 = 15 (二进制 1111)                               │   │
│  │  hash = 37 (二进制 100101)                                          │   │
│  │                                                                     │   │
│  │  37 & 15:                                                          │   │
│  │    100101                                                          │   │
│  │  & 001111                                                          │   │
│  │  ─────────                                                         │   │
│  │    000101 = 5                                                       │   │
│  │                                                                     │   │
│  │  所以 index = 5，等价于 37 % 16 = 5                                 │   │
│  │                                                                     │   │
│  │  位运算效率远高于取模！                                             │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 1.9 负载因子为什么是 0.75？

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    负载因子权衡                                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  【什么是负载因子？】                                                       │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  负载因子 = size / capacity                                          │   │
│  │                                                                     │   │
│  │  size: 当前元素数量                                                 │   │
│  │  capacity: 数组容量                                                 │   │
│  │                                                                     │   │
│  │  例: 容量 16，已存 12 个元素 → 负载因子 = 12/16 = 0.75              │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【为什么是 0.75？】                                                        │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  权衡结果（时间 vs 空间）                                           │   │
│  │                                                                     │   │
│  │  太小（如 0.5）:                                                    │   │
│  │  - ✅ 冲突少，查找快                                                │   │
│  │  - ❌ 数组利用率低，空间浪费                                        │   │
│  │  - 扩容频繁，性能开销大                                             │   │
│  │                                                                     │   │
│  │  太大（如 1.0）:                                                    │   │
│  │  - ✅ 数组利用率高                                                 │   │
│  │  - ❌ 冲突增多，链表变长，性能下降                                   │   │
│  │  - 最坏可能退化成 O(n)                                             │   │
│  │                                                                     │   │
│  │  0.75（平衡点）:                                                    │   │
│  │  - 时间和空间的最佳平衡                                             │   │
│  │  - 基于泊松分布的经验值                                             │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
│  【扩容触发】                                                               │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │  当 size / capacity > 0.75 时，数组扩容为原来的 2 倍              │   │
│  │                                                                     │   │
│  │  例: 16 → 32 → 64 → 128 ...                                        │   │
│  │                                                                     │   │
│  │  为什么是 2 倍？                                                    │   │
│  │  - 保持 2 的幂次方（位运算要求）                                     │   │
│  │  - 扩容时元素可通过 hash & newMask 重新分布                        │   │
│  └─────────────────────────────────────────────────────────────────────┘   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 二、常见面试题

### Q1: HashMap 为什么线程不安全？

**标准答案**：
1. **JDK 1.7**: 多线程扩容时，头插法可能导致死循环（链表成环）
2. **JDK 1.8**: 数据覆盖（多个线程同时 put 同一个 key）
3. **无同步机制**: 没有 synchronized 或 volatile 保证可见性

### Q2: HashMap 1.7 vs 1.8 有什么区别？

**标准答案**：
| 方面 | JDK 1.7 | JDK 1.8 |
|------|---------|---------|
| 数据结构 | 数组 + 链表 | 数组 + 链表 + 红黑树 |
| 插入方式 | 头插法 | 尾插法 |
| 扩容死循环 | 可能发生 | 解决了 |
| 红黑树阈值 | 无 | 链表 ≥8 且数组 ≥64 |

### Q3: 为什么重写 equals() 必须重写 hashCode()？

**标准答案**：
```java
// HashMap 通过 hashCode() 定位数组位置
// 通过 equals() 判断 key 是否相等

// 如果只重写 equals()，不重写 hashCode()
// 相等的对象可能 hash 值不同
// → 存储在不同位置
// → get() 时找不到！

// 正确做法：相等的对象必须有相同的 hash 值
```

### Q4: HashMap 的遍历方式有哪些？

**标准答案**：
```java
// 1. EntrySet 遍历（推荐）
for (Map.Entry<String, Integer> entry : map.entrySet()) {
    String key = entry.getKey();
    Integer value = entry.getValue();
}

// 2. KeySet 遍历
for (String key : map.keySet()) {
    Integer value = map.get(key);  // 多一次查找
}

// 3. Lambda 遍历（JDK 8+）
map.forEach((key, value) -> {
    // ...
});

// 4. Iterator 遍历（可安全删除）
Iterator<Map.Entry<String, Integer>> it = map.entrySet().iterator();
while (it.hasNext()) {
    Map.Entry<String, Integer> entry = it.next();
    if (entry.getValue() == null) {
        it.remove();  // 安全删除
    }
}
```

## 三、常见误区

### ❌ 误区1：HashMap 初始容量越大越好

```
正确理解：
初始容量过大会导致:
- 内存浪费
- 遍历效率降低

应该根据预计元素数合理设置:
initialCapacity = 预计元素数 / 0.75 + 1
```

### ❌ 误区2：HashMap 找不到元素就返回 null

```
正确理解：
HashMap 允许存储 null 值！

map.get("key") 返回 null 有两种情况:
1. key 不存在
2. key 存在，但对应的 value 是 null

区分方法:
if (map.containsKey("key") && map.get("key") == null) {
    // value 是 null
} else if (!map.containsKey("key")) {
    // key 不存在
}
或使用:
Integer value = map.getOrDefault("key", -1);
```

### ❌ 误区3：HashMap 遍历顺序固定

```
正确理解：
HashMap 不保证遍历顺序！
遍历顺序可能与插入顺序不同

如果需要有序:
- LinkedHashMap（插入顺序或访问顺序）
- TreeMap（按 key 排序）
```
