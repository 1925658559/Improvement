# Day 12: 子集（回溯进阶）+ MySQL 索引原理

## 🔥 算法热身

**题目**: LeetCode 78 - 子集

**题目描述**:
给你一个整数数组 `nums`，数组中的元素**互不相同**。返回该数组所有可能的子集（幂集）。

**示例**:
```
输入：nums = [1,2,3]
输出：[[],[1],[2],[1,2],[3],[1,3],[2,3],[1,2,3]]
```

---

## 📝 算法解法详解

### 核心思想：回溯算法（Backtracking）

**子集 vs 全排列的关键区别**：

| 特性 | 全排列 | 子集 |
|------|--------|------|
| **长度** | 固定（都是 n 个元素） | 不固定（0 到 n 个元素） |
| **顺序** | 有序性重要（[1,2,3] ≠ [3,2,1]） | 顺序无关（[1,2] = [2,1]） |
| **递归终止** | `path.size() == n` | 每个节点都加入结果集 |
| **去重方式** | `boolean[] used` | `start` 索引 |

---

### 解法：回溯 + start 索引

**核心思想**：
1. **每个节点都是一个有效的子集**（包括空集）
2. 使用 `start` 索引控制选择范围，避免重复
3. 递归时传 `i+1`，保证后续只选择当前元素之后的元素

**完整代码**：
```java
public List<List<Integer>> subsets(int[] nums) {
    List<List<Integer>> result = new ArrayList<>();
    List<Integer> path = new ArrayList<>();

    // 从索引 0 开始回溯
    backtrack(nums, 0, path, result);

    return result;
}

private void backtrack(int[] nums, int start,
                      List<Integer> path, List<List<Integer>> result) {
    // 1. 把当前 path 加入结果集（每个节点都是一个有效的子集）
    result.add(new ArrayList<>(path));

    // 2. 遍历：从 start 开始，到最后一个元素
    for (int i = start; i < nums.length; i++) {
        // 3. 做选择：选择 nums[i]
        path.add(nums[i]);

        // 4. 递归：从 i+1 开始（避免重复选择）
        backtrack(nums, i + 1, path, result);

        // 5. 撤销选择（回溯）
        path.remove(path.size() - 1);
    }
}
```

**复杂度分析**:
- 时间复杂度: **O(n × 2ⁿ)** - 2ⁿ 个子集，每个需要 O(n) 时间生成
- 空间复杂度: **O(n × 2ⁿ)** - 结果集存储 2ⁿ 个子集
- 递归深度: O(n)

### 🔑 关键知识点

#### 1. 为什么第一行就要 `result.add(path)`？

**原因**：
- 子集的长度可以是 0 到 n
- **每个节点的 path 都是一个有效的子集**
- 包括空集 `[]`（第一次调用时 path 是空的）

**对比全排列**：
- 全排列：只有叶子节点才是有效结果（`path.size() == n`）
- 子集：所有节点都是有效结果

---

#### 2. 为什么用 `start` 索引而不是 `used[]`？

**原因**：
- 子集的顺序无关：`[1,2]` 和 `[2,1]` 是同一个子集
- 使用 `start` 索引可以保证**只选择当前元素之后的元素**
- 这样就避免了重复

**示例**：
```
如果已经选择了 1：
- 下次只能从 [2, 3] 中选择（start = 1）
- 不会再选择 1（避免了 [2,1] 这种重复）
```

---

#### 3. 为什么递归传 `i+1` 而不是 `start+1`？

**错误代码**：
```java
// ❌ 错误：传 start+1
backtrack(nums, start + 1, path, result);
```

**正确代码**：
```java
// ✅ 正确：传 i+1
backtrack(nums, i + 1, path, result);
```

**原因**：
- 当前选择了 `nums[i]`
- 下次应该从 `i+1` 开始选择（跳过已选择的元素）
- 如果传 `start+1`，可能会重复选择元素

---

### ⚠️ 常见错误

**1. 忘记创建新的 ArrayList**
```java
// ❌ 错误
result.add(path);

// ✅ 正确
result.add(new ArrayList<>(path));
```

**原因**：`path` 是引用，会不断变化，必须创建副本。

---

**2. 递归传错参数**
```java
// ❌ 错误：传 start+1
backtrack(nums, start + 1, path, result);

// ✅ 正确：传 i+1
backtrack(nums, i + 1, path, result);
```

---

**3. 忘记回溯（撤销选择）**
```java
// ❌ 错误：忘记撤销选择
path.add(nums[i]);
backtrack(nums, i + 1, path, result);
// 忘记了 path.remove()

// ✅ 正确：必须撤销选择
path.add(nums[i]);
backtrack(nums, i + 1, path, result);
path.remove(path.size() - 1);  // 撤销
```

---

**4. 循环条件错误**
```java
// ❌ 错误：从 0 开始
for (int i = 0; i < nums.length; i++) {
    // 会产生重复
}

// ✅ 正确：从 start 开始
for (int i = start; i < nums.length; i++) {
    // 避免重复
}
```

## 🧠 核心技术：MySQL 索引原理

### 一、为什么索引快？

**场景**：假设有一张用户表，有 **100 万条数据**

```sql
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    age INT,
    city VARCHAR(50)
);
```

**查询对比**：

| 查询方式 | 时间复杂度 | 查询次数（100万数据） | 说明 |
|---------|-----------|---------------------|------|
| **全表扫描**（无索引） | O(n) | ~1,000,000 次 | 一条一条遍历 |
| **索引查询**（B+树） | O(log n) | ~3 次 | 树形结构查找 |

**性能提升**：**33 万倍**！

---

### 二、B+树 vs 二叉树

**为什么 MySQL 使用 B+树而不是二叉树？**

**对比表格**：

| 特性 | 二叉树 | B+树 |
|------|--------|------|
| **每个节点的子节点数** | 最多 2 个 | 可以有 1000+ 个 |
| **树高（100万数据）** | log₂(10⁶) ≈ 20 | log₁₀₀₀(10⁶) ≈ 2~3 |
| **查询次数（page read）** | ~20 次 | ~3 次 |
| **随机 I/O 耗时**（假设 1ms/次） | ~20ms | ~3ms |
| **性能差距** | - | **6~7 倍** |

**关键结论**：
- B+树的**分支因子（fanout）**大 → 树更矮 → 随机 I/O 次数少
- 影响 fanout 的关键：`page size` 与 `key + pointer` 的大小

---

### 三、B+树的结构

**ASCII 图示**：

```
B+树结构（每个节点存储 1000 个键）：

第 1 层（根节点）：
┌─────────────────────────────────────┐
│ 1 | 1000 | 2000 | ... | 999000     │  ← 1 个节点，1000 个键
└─────────────────────────────────────┘
         ↓         ↓         ↓
第 2 层（中间节点）：
┌──────┐  ┌──────┐  ┌──────┐
│ ...  │  │ ...  │  │ ...  │  ← 1000 个节点，每个 1000 个键
└──────┘  └──────┘  └──────┘
    ↓         ↓         ↓
第 3 层（叶子节点）：
┌──────┐  ┌──────┐  ┌──────┐
│ data │←→│ data │←→│ data │  ← 1,000,000 条数据（双向链表）
└──────┘  └──────┘  └──────┘

查询过程：
1. 访问第 1 层（根节点）→ 找到目标范围
2. 访问第 2 层（中间节点）→ 找到目标范围
3. 访问第 3 层（叶子节点）→ 找到目标数据

总共查询次数 = 3 次！
```

---

### 四、叶子节点为什么要用链表？

**原因**：**为了范围查询**

**示例**：
```sql
SELECT * FROM user WHERE age BETWEEN 20 AND 30;
```

**查询过程**：
1. 通过 B+树找到第一个满足条件的叶子节点（age=20）
2. 沿着叶子链表顺序扫描，直到 age>30

**关键优势**：
- 把**多次随机 I/O** 变成 **顺序 I/O**
- 顺序 I/O 更容易被 **read-ahead / OS page cache / buffer pool** 命中
- 吞吐更高、延迟更低

---

### 五、聚簇索引 vs 二级索引

#### 1. 聚簇索引（Clustered Index）

**定义**：在 InnoDB 里，**主键 B+树的叶子节点存储整行 row data**

**ASCII 图示**：

```
主键索引（聚簇索引）：

         [10 | 20 | 30]  ← 根节点（存 key）
         /      |      \
    [1..9]  [11..19]  [21..29]  ← 中间节点
      /        |          \
   [row]    [row]       [row]  ← 叶子节点（存整行数据）
   id=1     id=11       id=21
   name=..  name=..     name=..
   age=..   age=..      age=..
```

---

#### 2. 二级索引（Secondary Index）

**定义**：除了主键索引之外的所有索引，叶子节点存储 **(secondary key + 主键 id)**

**示例**：
```sql
CREATE TABLE user (
  id   BIGINT PRIMARY KEY,
  name VARCHAR(50),
  age  INT,
  KEY idx_name(name)  -- 二级索引
);
```

**ASCII 图示**：

```
二级索引（idx_name）：

         ['李' | '王' | '赵']  ← 根节点
         /       |        \
    ['张'..]  ['李'..]  ['王'..]  ← 中间节点
      /          |           \
   [name='张三', id=1]  ← 叶子节点（存 name + 主键 id）
   [name='张四', id=5]
   [name='张五', id=9]
```

---

### 六、回表（Back to Table）

**场景**：
```sql
SELECT * FROM user WHERE name = '张三';
```

**查询过程**：
1. 在二级索引 `idx_name` 中找到 `name='张三'`，得到 `id=1`
2. 拿着 `id=1` 去聚簇索引（主键索引）中查找整行数据
3. 返回整行数据

**这个过程叫"回表"（Back to Table / Bookmark Lookup）**

**ASCII 图示**：

```
步骤 1：在二级索引中查找
idx_name: [name='张三', id=1]  ← 找到了！

步骤 2：回表（去聚簇索引查找）
PRIMARY: [id=1, name='张三', age=25, city='北京']  ← 找到整行数据
```

---

### 七、覆盖索引（Covering Index）

**定义**：查询的列完全由索引提供，**不需要回表**

**示例**：
```sql
-- 查询 1：需要回表
SELECT * FROM user WHERE name = '张三';
-- 需要：name, id, age, city
-- 索引提供：name, id
-- 缺少：age, city → 需要回表

-- 查询 2：不需要回表（覆盖索引）
SELECT id, name FROM user WHERE name = '张三';
-- 需要：id, name
-- 索引提供：name, id
-- 完全覆盖 → 不需要回表
```

**执行计划**：
```
EXPLAIN SELECT id, name FROM user WHERE name = '张三';

Extra: Using index  ← 覆盖索引的标志
```

### 八、常见面试题

#### Q1: 为什么 MySQL 使用 B+树而不是二叉树？

**标准答案**：
B+树的**分支因子（fanout）**大，树更矮，随机 I/O 次数少。

**详细解释**：
- 二叉树：每个节点最多 2 个子节点，树高 log₂(n) ≈ 20（100万数据）
- B+树：每个节点可以有 1000+ 个子节点，树高 log₁₀₀₀(n) ≈ 3（100万数据）
- 查询次数：20 次 vs 3 次，性能提升 6~7 倍

---

#### Q2: B+树的叶子节点为什么要用链表串起来？

**标准答案**：
为了**范围查询**。

**详细解释**：
- 范围查询（`BETWEEN`、`>`、`<`）需要扫描多个叶子节点
- 叶子链表把**多次随机 I/O** 变成 **顺序 I/O**
- 顺序 I/O 更容易被缓存命中，性能更好

---

#### Q3: 什么是聚簇索引？什么是二级索引？

**标准答案**：
- **聚簇索引**：主键 B+树，叶子节点存储**整行 row data**
- **二级索引**：非主键索引，叶子节点存储 **(secondary key + 主键 id)**

---

#### Q4: 什么是回表？如何避免回表？

**标准答案**：
- **回表**：通过二级索引找到主键后，再去聚簇索引查找整行数据
- **避免回表**：使用**覆盖索引**（查询列完全由索引提供）

**示例**：
```sql
-- 需要回表
SELECT * FROM user WHERE name = '张三';

-- 不需要回表（覆盖索引）
SELECT id, name FROM user WHERE name = '张三';
```

---

#### Q5: 为什么二级索引不直接存储整行数据，而是存储主键？

**标准答案**：
1. **减少数据冗余**：如果每个二级索引都存整行，数据会重复多份
2. **降低更新成本**：更新一行数据时，只需要更新聚簇索引，不需要更新所有二级索引
3. **需要整行时再回表**：通过主键去聚簇索引查找

---

#### Q6: 一次随机磁盘 I/O 大概需要多少时间？

**标准答案**：
- **NVMe SSD**：0.05–0.2 ms
- **普通 SSD**：0.5–2 ms
- **机械硬盘 HDD**：5–15 ms

---

#### Q7: 为什么 B+树的分支因子可以很大？

**标准答案**：
因为 **internal node（非叶子节点）只存储 key + pointer**，entry 很小。

**详细解释**：
- 假设 page size = 16KB
- 假设 key + pointer = 16 bytes
- 那么一个 page 可以存储：16KB / 16B = 1000 个 entry
- 所以分支因子 ≈ 1000

### 九、常见误区

#### ❌ 误区1：索引越多越好

```
正确理解：
索引会占用存储空间，并且会降低写操作（INSERT/UPDATE/DELETE）的性能。
应该根据查询需求合理创建索引。
```

---

#### ❌ 误区2：二级索引的叶子节点存储整行数据

```
正确理解：
InnoDB 的二级索引叶子节点存储的是 (secondary key + 主键 id)。
需要整行数据时，通过主键去聚簇索引查找（回表）。
```

---

#### ❌ 误区3：覆盖索引一定比回表快

```
正确理解：
大部分情况下，覆盖索引比回表快。
但如果查询需要的列很多，创建覆盖索引的成本可能很高。
需要权衡查询性能和存储成本。
```
